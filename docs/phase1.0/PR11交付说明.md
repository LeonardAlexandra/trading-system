# PR11 交付说明（可一键复制）

---

## 1）实现摘要

在既有架构与 Phase1.0 约束下完成 PR11：多 strategy_id 支持 + 策略级配置解析 + 风控/执行按 strategy_id 隔离 + CONFIG_SNAPSHOT 增强。

- 多 strategy_id：Webhook payload 必须带 strategy_id；未提供 → 422 MISSING_STRATEGY_ID；不存在/禁用 → 422 STRATEGY_NOT_FOUND / STRATEGY_DISABLED。
- 策略级配置：AppConfig.strategies（StrategyEntryConfig）、_from_dict 解析 strategies，向后兼容「仅顶层 strategy_id」；启动期 validate 校验 strategy 唯一性与合并后 execution/risk。
- StrategyConfigResolver：resolve(app_config, strategy_id) 返回 ResolvedStrategyConfig（execution/risk/exchange + strategy_config_fingerprint）；运行期只读 app_config。
- 执行与风控绑定 strategy_id：DecisionOrderMap/Execution/Trade/Order/execution_events 沿用既有 strategy_id；RiskManager 冷却、同向去重、仓位均按 strategy_id 隔离；RiskManager.check(decision, risk_config_override) 支持策略级风控。
- CONFIG_SNAPSHOT 增强：message 含 strategy_id、strategy_config_fingerprint；白名单、无 secret、MAX_SNAPSHOT_BYTES=4096、超限截断。
- Position 按 strategy_id 隔离：主键改为 (strategy_id, symbol)，迁移 007；PositionRepository 所有读写带 strategy_id。
- Paper 模式：仍使用 PaperExchangeAdapter，未引入真实交易所。

## 5）对系统长期演进的影响评估

- 多策略与实盘：当前按 strategy_id 隔离配置与状态，为多策略与后续实盘切换预留接口（仅换 adapter，不改决策/风控/审计边界）。
- 审计与可观测：CONFIG_SNAPSHOT 带 strategy_id 与 fingerprint，便于按策略追溯配置版本与变更。
- 配置与风控扩展：strategies + StrategyEntryConfig 可继续扩展 profile 引用、更多 override 字段，在保持 fail-fast 与白名单前提下演进。

## C. 系统风险与回归影响评估（逐条回答）

**1）本次 PR11 是否改变了：**

- 现有幂等语义？否。decision_id 仍为幂等主键；两段式占位（RESERVED → SUBMITTING → FILLED/FAILED）未变。
- ExecutionEngine 的事务边界？否。事务 A（抢占）/ 交易所调用 / 事务 B（落库）边界不变；仅在事务内按 strategy_id 解析配置与写快照、按 strategy_id 更新仓位与风控状态。
- RiskManager 的原有判定逻辑？逻辑不变（冷却、同向去重、仓位上限、单笔上限）；仅改为「每次 check 可传入策略级 risk_config_override」且仓位按 strategy_id 查询，判定规则本身未改。

**2）本次修改可能引入的新风险点：**

- 并发风险：单实例、decision_id 幂等与抢占未变，未新增跨请求共享可变状态；多策略并发仍由同一套幂等与事务保证。
- 配置复杂度风险：strategies + override 增加配置面；通过启动期校验与测试降低错误配置上线概率。
- 状态隔离遗漏风险：若未来新增「按 symbol 或全局」的状态表/缓存，可能漏带 strategy_id；当前 Position/RiskState/DecisionOrderMap 已全部按 strategy_id 隔离，并在测试中覆盖。

**3）如果需要回滚 PR11：**

- 是否可以通过配置回滚？仅能通过「只保留单策略、且该策略使用原 strategy_id」的方式在行为上近似回滚；多策略能力需代码回滚。
- 是否涉及数据库 schema 变更？是。007 增加 positions 的 strategy_id 并改主键；回滚需执行 007 的 downgrade（会丢弃非 default 策略的 positions 数据）。
- 是否会影响已存在的 execution_events 审计链？不影响。已写入的 execution_events 不变；回滚后新事件不再带 strategy_id/fingerprint（若用旧版快照逻辑），历史链仍可读。
