# PR16：灰度演练 + 强门禁 — 工程级校验证据包

## 1）变更摘要与边界

### 变更摘要
- **目标 1**：多重 Live 门禁（execution.live_enabled 已有；新增 allow_real_trading、live_allowlist_accounts、live_confirm_token）；仅当适配器为“真实交易所”（is_real_trading=True，如 OKX）时校验，Paper/DryRun 不校验。
- **目标 2**：参数精度与数量校验（qty>0、精度可配置、禁止 NaN/inf、市价单最大名义价值可选）；校验失败写 ORDER_REJECTED，不发起 HTTP。
- **目标 3**：执行模式 execution.mode（DRY_RUN | DEMO_LIVE_REHEARSAL）；演练模式下更严限频/断路器，event message 标记 rehearsal=true。
- **目标 4**：事故演练测试（OKX 5xx → 重试 → 断路器打开 → 全局拒单）；参数错误 → 本地拒绝 → 不触发 HTTP。

### 边界
- **不破坏**：PR14a live_enabled 门禁、PR15b OKX Demo 闭环、重试策略门禁、事件体系统一、默认测试离线可跑。
- **不引入**：真实实盘交易；减少或绕过既有门禁；参数校验完全交给交易所；破坏 execution_events 语义；泄露 secret/header/签名。

## 4）事故演练场景与回滚说明

### 场景 1：OKX 连续 5xx → 重试 → 断路器打开 → 全局拒单
- **复现步骤**：配置 circuit_breaker_threshold=2、max_attempts=1；FakeOkx 返回 code=500；执行 2 笔决策均失败；第三笔执行时断路器已开。
- **预期 execution_events 序列**：  
  第一/二笔：CLAIMED → … → ORDER_SUBMIT_STARTED → OKX_HTTP_CREATE_ORDER / ORDER_SUBMIT_FAILED；第二笔含 CIRCUIT_OPENED。  
  第三笔：CLAIMED → … → FINAL_FAILED，reason_code=CIRCUIT_OPEN（不发起 HTTP）。
- **人工回滚/恢复**：等待 circuit_breaker_open_seconds 后断路器自动半开；或清空 circuit_breaker_state 表中对应 account_id 记录（需运维脚本）；恢复 OKX 服务后重试。

### 场景 2：参数精度错误 → 本地校验拒绝 → 不触发 HTTP
- **复现步骤**：配置 order_qty_precision=4；决策 quantity=1.12345；执行 execute_one。
- **预期**：ORDER_REJECTED，reason_code=ORDER_QTY_PRECISION_EXCEEDED；无 OKX_HTTP_CREATE_ORDER。
- **人工回滚/恢复**：修正策略或配置的 qty/精度后重新发起信号；无需交易所侧操作。

**证据**：`tests/integration/test_pr16_incident_rehearsal.py::test_pr16_okx_5xx_then_circuit_open_rejects`；`tests/integration/test_pr16_param_validation.py::test_qty_precision_exceeded_rejected`。

- **幂等**：decision_id = client_order_id，抢占与落库不变。
- **事务边界**：事务 A 抢占、事务 B 落库，未改变。

- **Secret**：未写入 log/events/snapshot；OKX_HTTP_* 仅含 action/http_status/okx_code/request_id/attempt。
- **PR15b**：OKX Demo 闭环测试在满足 PR16 门禁（allow_real_trading=true、LIVE_CONFIRM_TOKEN）下全部通过。

## 8）风险清单与残余风险声明
