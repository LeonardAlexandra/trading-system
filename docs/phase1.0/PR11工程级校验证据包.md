- **是否改变了以下行为之一？**  
  - **幂等主键语义**：**否**。decision_id 仍为幂等主键；两段式占位（RESERVED → SUBMITTING → FILLED/FAILED）未变；resolve 失败仅多一条“抢占后立即 FAILED”的路径。  
  - **ExecutionEngine 事务边界**：**否**。resolve 失败在“事务 A 抢占成功之后、下单之前”return，不进入交易所调用与事务 B；FAILED 更新与事件写入仍在同一 session/事务内。

  - **CONFIG_SNAPSHOT 内容或大小策略**：**否**。仍为白名单、无 secret、MAX_SNAPSHOT_BYTES=4096、超限截断；仅增加 strategy_id、strategy_config_fingerprint，且 resolve 失败时不写 CONFIG_SNAPSHOT。  

**证据来源**：上述结论来自当前代码逻辑（execution_engine、risk_manager、app_config、snapshot、007 migration）与全部 45 个测试通过结果。
