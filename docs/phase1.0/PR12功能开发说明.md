# PR12 功能开发说明

## 做了什么

### 0. 工程级整改（消除阻塞 PR13 的结构性缺陷）

- **OrderManager 行为进入 execution_events 审计链**：cancel_order / sync_order_status 成功后追加 ORDER_CANCELLED / ORDER_SYNCED 事件；事件含 order_id、exchange_order_id、strategy_id（message）、原状态→新状态、reason/source；OrderManager 注入 ExecutionEventRepository，不在此内 commit。
- **ExecutionEngine 与 OrderManager 职责边界**：在 `order_manager.py` 模块注释中约定：可取消状态（SUBMITTED/PENDING/PARTIAL）、cancel/sync 后 Engine 不改写 orders 表、状态优先级 FILLED > CANCELLED > FAILED；新增 `tests/integration/test_order_manager_audit.py`：Engine 执行 → 插入 orders → OrderManager cancel/sync → 断言 execution_events 顺序与最终订单状态正确。

### 1. ExchangeAdapter 扩展（`src/execution/exchange_adapter.py`）

- 新增 **GetOrderResult** 数据类：`exchange_order_id`、`status`、`filled_qty`、`avg_price`、`raw`。
- 抽象方法：
  - `get_order(exchange_order_id: str, symbol: str) -> GetOrderResult`
  - `cancel_order(exchange_order_id: str) -> bool`
- **PaperExchangeAdapter**：
  - 内存 `_orders` 存储订单状态；
  - `create_order` 写入、`get_order` 读取、`cancel_order` 将未成交订单置为 CANCELLED；
  - 已成交订单调用 `cancel_order` 返回 False。

### 2. OrdersRepository 扩展（`src/repositories/orders_repo.py`）

- 新增 **update(order: Order) -> Order**：内部 `session.add(order)`，不在此处 commit，由调用方管理 session。

### 3. OrderManager（`src/execution/order_manager.py`，新建 + 整改）

- **构造函数**：`__init__(exchange_adapter, order_repo, event_repo: ExecutionEventRepository)`。
- **get_order(order_id: str) -> Order | None**：仅读库，`order_repo.get_by_local_order_id(order_id)`。
- **cancel_order(order_id: str, reason: str, *, source: str = "user") -> bool**：查库 → 调 adapter.cancel_order → 成功则本地 status=CANCELLED、update，并 **append_event(ORDER_CANCELLED)**（含 order_id、exchange_order_id、strategy_id、old_status→new_status、reason/source）；不 commit。
- **sync_order_status(order_id: str, *, source: str = "system") -> Order**：查库 → adapter.get_order → 更新 status/filled_quantity、update，并 **append_event(ORDER_SYNCED)**；订单不存在抛 `ValueError`；不 commit。

### 4. 订单状态常量（`src/common/order_status.py`）

- 新增 **ORDER_STATUS_CANCELLED = "CANCELLED"**，供取消逻辑使用。

### 5. 事件类型（`src/common/event_types.py`）

- **ORDER_CANCELLED** / **ORDER_SYNCED**：OrderManager cancel/sync 审计事件类型。

### 6. 测试（`tests/execution/test_order_manager.py` + `tests/integration/test_order_manager_audit.py`）

- 单元：7 个用例，其中 cancel/sync 成功用例断言 execution_events 含 ORDER_CANCELLED/ORDER_SYNCED 及 message、reason_code。
- 集成：2 个用例（test_engine_then_order_manager_cancel_events_and_order_state、test_engine_then_order_manager_sync_events_and_order_state），覆盖 Engine 执行 → 插入 orders → OrderManager cancel/sync → events 顺序与订单状态正确。

## 关键设计决策

- **order_id 语义**：OrderManager 的 `order_id` 一律指本地主键 `Order.order_id`；与交易所交互用 `Order.exchange_order_id`，由 Manager 内部转换。
- **不在此内 commit**：OrderManager 与 OrdersRepository.update 均不 commit，由调用方统一管理 session，与现有 ExecutionEngine/事务边界一致。
- **Paper 内存存储**：Paper 用内存 dict 维护订单状态，使 get/cancel/sync 可测；进程重启丢失，符合 Phase1.0 纸面交易所范围。
- **sync 不存在即抛错**：`sync_order_status` 约定返回 `Order`，订单不存在时抛 `ValueError`，避免隐式 None。
