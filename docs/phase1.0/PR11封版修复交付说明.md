# PR11 封版修复交付说明（可一键复制）

---

## 1）修复实现摘要

### 隐患 1：ExecutionEngine 的 resolve fallback 行为

- 原问题：strategy resolve 失败时回退到全局 app_config 继续执行，可能导致「策略 A 的信号用策略 B 的配置成交」，违反“运行期强假设 + 明确失败”。
- 修复方式：
  - 在 execute_one 中，当 app_config 且 strategy_id 存在时，只做一次 resolve(app_config, strategy_id)。
  - 若抛出 StrategyConfigResolverError：取 e.reason_code（如 STRATEGY_NOT_FOUND / STRATEGY_DISABLED）；其他异常用 CONFIG_UNAVAILABLE。
  - 失败时：写 execution_events（event_type=FINAL_FAILED，reason_code + message）、decision_order_map 更新为 FAILED 并设置 last_error，立即 return，不写 CONFIG_SNAPSHOT、不风控、不下单、不更新仓位。
  - 成功时：用同一 resolved 写 CONFIG_SNAPSHOT 并作为 risk_config_override，不再对 risk 做第二次 resolve。
- 修复后行为：resolve 失败 → 该 decision 明确 FAILED、有 reason_code 的审计事件、不成交、不更新仓位；无任何 fallback 执行。

### 隐患 2：多策略 position_snapshot 不可逆性未在系统中显式承认

- 原问题：position 已按 (strategy_id, symbol) 升级，downgrade 会丢非 default 策略数据，此前仅在文档中说明。
- 修复方式：
  1. 启动期 fail-fast：在 AppConfig.validate() 中，若 len(strategies) > 1 且环境变量 ALLOW_POSITION_SCHEMA_DOWNGRADE=true，则抛出 ConfigValidationError(MULTI_STRATEGY_POSITION_DOWNGRADE_FORBIDDEN, ...)，启动失败。
  2. 迁移 007 显式约束：在 007 中定义 MULTI_STRATEGY_SCHEMA_VERSION = 2，并在 downgrade() 开头检查 os.environ.get("ALLOW_DATA_LOSS") == "true"，否则 raise RuntimeError(...)，禁止未显式确认的降级。
- 修复后行为：多策略下不允许通过 ALLOW_POSITION_SCHEMA_DOWNGRADE 规避约束；执行 007 的 downgrade 必须设置 ALLOW_DATA_LOSS=true，否则直接报错。

## 4）回归影响分析

**1）本次修复是否影响：**

- 幂等语义？否。decision_id 仍为幂等主键；resolve 失败仅多一条“抢占后立即 FAILED”的路径，不改变两段式占位与重试语义。
- 事务边界？否。resolve 失败在“事务 A 抢占成功之后、下单之前”返回，不进入交易所调用与事务 B；FAILED 更新与事件写入仍在同一 session/事务内。
- 风控判定逻辑？否。风控规则未改；仅在“有 app_config 且 strategy_id 且 resolve 成功”时使用 resolved.risk 作为 override，否则与之前一致（无 override）。

**2）是否引入新的行为不兼容、运维复杂度、配置风险？**

- 行为不兼容：有且仅有一类：此前“decision 的 strategy_id 不在 app_config.strategies 中却仍能执行”的隐式 fallback 行为被移除，现会明确 FAILED。这是封版要求的正确行为。
- 运维复杂度：多策略环境下需注意不要设置 ALLOW_POSITION_SCHEMA_DOWNGRADE=true；执行 007 的 downgrade 时必须设置 ALLOW_DATA_LOSS=true。均为显式、一次性操作。
- 配置风险：无新增配置项；仅对既有 strategies 与两个环境变量组合做约束。

**3）如果需要回滚本修复：**

- 是否只需代码回滚？是。恢复 ExecutionEngine 的 resolve fallback、去掉 AppConfig 中多策略约束、去掉 007 中 ALLOW_DATA_LOSS 检查即可。
- 是否涉及数据迁移？否。未改表结构，未改已有数据。

## 5）残余风险（如有）

- 历史 decision：若线上已有 strategy_id 不在当前 strategies 中的 RESERVED 决策，修复后会被执行时 resolve 失败并标为 FAILED。建议上线前确认无此类“孤儿”决策，或接受其被标记为 FAILED。
- 007 downgrade 使用方式：运维需知执行 alembic downgrade -1 到 006 前必须设置 ALLOW_DATA_LOSS=true，否则会 RuntimeError；可在运维文档中明确写出该步骤。

除上述两点外，无其他残余风险；幂等、事务边界与风控逻辑均未改变，仅消除了 resolve fallback 与多策略 position 不可逆性的隐性行为。
