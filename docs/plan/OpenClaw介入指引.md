# OpenClaw 介入指引（首次加入必读）

**对象**：OpenClaw（与 Cursor 共同开发本项目的另一 agent）  
**目标**：介入后迅速知道「如何与 Cursor 协作、当前该做哪一模块、按什么规则交付」，无需用户逐项说明。

---

## 一、你与 Cursor 的关系（一句话）

- 你与 **Cursor** 按模块分工：每个模块**一方为开发者**（写代码与证据包初稿），**另一方为审计者**（只读代码与证据包，对照交付包审查，通过则签字/更新分工表）。分包完成，互不写同一模块代码，故不互相干扰。
- **用户**只做验收与关键决策，不负责「谁开发、谁审计」的日常指派；**角色与下一模块**由分工表 + 协作方案中的约定决定。

---

## 二、介入后立刻要读的 3 个东西（按顺序）

| 顺序 | 读什么 | 路径 | 用来知道 |
|------|--------|------|----------|
| 1 | 协作方案（目的、用户角色、双 agent 自协作、分工约定） | `docs/plan/Cursor与OpenClaw协作方案.md` | 为什么这样协作、你要遵守的原则、交接方式 |
| 2 | 当前活跃模块与分工表 | `docs/plan/当前活跃模块与分工表.md` | **当前 Phase、当前/下一模块、你是开发者还是审计者、证据包路径** |
| 3 | 本 Phase 的交付包（你被指派的那个模块的 Goal / Scope / Acceptance） | 见下「三、交付包路径」 | 本模块要做啥、能改哪些路径、验收标准 |

**若 `当前活跃模块与分工表.md` 不存在**：按协作方案中的「未完成任务清单」与「开发/审计约定」，取**当前 Phase 里尚未出现证据包的第一个模块**作为待做模块，并依约定确定你本模块是**开发者**还是**审计者**（Phase 2.0 剩余顺序：C3 → C4 → C5 → D1～D5）。

## 三、交付包路径（按 Phase 查你的模块）

- **Phase 2.0**：`docs/plan/Phase2.0_模块化开发交付包.md`（模块 A1/A2/C1～C5、D1～D5）
- **Phase 2.1**：`docs/plan/Phase2.1_模块化开发交付包.md`（模块 A1～A3、B1～B5、C1～C4、D1～D6）
- **Phase 2.2**：`docs/plan/Phase2.2_模块化开发交付包.md`（模块 A1/A2、B1～B4、D1～D4）

在对应交付包内用**模块编号**（如 C3、C4）搜索，找到「开发范围」「强制约束」「验收口径」「本模块完成后必须回传的证据包」等小节，**严格按该范围实现，不得改 scope 外文件**。

---

## 四、证据包：你必须交付的东西

每个模块做完后，**必须**生成或更新一份**证据包**（Markdown），否则视为未完成。证据包须包含：

- 模块名称与目标  
- 本次修改/新增的文件清单  
- 实际执行的命令（逐条）  
- 命令的**真实输出**（可引用 `docs/runlogs/xxx.txt`，禁止伪造或省略关键输出）  
- 验收结论（与交付包中 Acceptance Criteria 逐条对照）

证据包命名与位置示例：`docs/Phase2.0_C3_模块证据包.md`（与现有 Phase2.0_A1、C1、C2 等一致）。  
项目强制规则：`.cursor/rules/phase12-module-evidence.mdc`（证据包结构、禁止跨模块修改等）。

---

## 五、做完本步后你必须做的事（按角色）

- **若你是开发者**：  
  1. 写好证据包初稿并自检（命令真实执行过、输出已记录、与验收口径逐条对照）。  
  2. 更新分工表：将本模块状态设为「待审计」，注明审计者（Cursor 或 OpenClaw）、证据包路径。  
- **若你是审计者**：  
  1. 只读该模块代码与证据包，对照交付包逐项审查；可复现关键命令。  
  2. **通过**：更新分工表（本模块已通过审计、下一模块与下一开发者），或**明确回传通过结论**给 Cursor/用户。**你的通过结论是 Cursor 进入下一模块的唯一条件**——未收到你的通过前，Cursor 不得开始下一模块的开发、测试或修改。  
  3. **不通过**：写明退回理由（引用交付包条目），由 Cursor 返工；返工后再次提交审计，通过后同上。  
这样 Cursor 可主动问询你是否通过，或等待你更新分工表/回传结论后，再进入下一模块；**无需用户指派**。

---

## 六、禁止事项（与 Cursor 协作时）

- **开发者**：**禁止**与审计者同时改同一模块代码；**禁止**修改当前模块 **scope 外**的代码或文档；**禁止**未真实执行命令却在证据包中声称已执行。  
- **审计者**：**禁止**修改该模块实现代码；**禁止**擅自放宽验收标准。  
- **双方**：**禁止**跳过模块或 Phase 顺序（2.0 → 2.1 → 2.2）；**禁止**擅自扩展或改写需求与模块边界；有歧义或想改范围时由**用户**决定。

## 七、快速自检（介入后问自己）

- [ ] 我已读 `Cursor与OpenClaw协作方案.md` 中的目的、连接方式、开发/审计分工、双 agent 自协作。  
- [ ] 我已读 `当前活跃模块与分工表.md`（若存在），确定「当前我是开发者还是审计者、对应模块与证据包路径」或按顺序自取。  
- [ ] 我已在该 Phase 的交付包中找到该模块的 Goal、Scope、Acceptance 与证据包要求。  
- [ ] 若为开发者：只改该模块 scope 内路径，完成后写证据包初稿并更新分工表（待审计）。若为审计者：只读不写代码，审查后更新分工表或退回。

若以上均为是，即可开始；若有不确定，以**交付包 + 协作方案**为准，不要自行发明规则。
