## 二、模块详细接口规范

### 2.1 SignalReceiver（信号接收器）

#### 职责（必须做什么）
- 接收来自 TradingView 的 Webhook 请求
- 验证请求签名和来源
- 将原始信号转换为内部标准格式
- 记录信号接收时间戳
- 将信号传递给 SignalParser

#### 禁止职责（绝对不允许做什么）
- ❌ 解析信号内容（由 SignalParser 负责）
- ❌ 执行任何交易逻辑
- ❌ 修改信号内容
- ❌ 缓存信号（必须实时传递）
- ❌ 进行风控检查

#### 输入数据结构
```python
# HTTP Webhook 请求
WebhookRequest {
    method: "POST"
    headers: {
        "X-TradingView-Signature": str  # 签名验证
        "Content-Type": "application/json"
    }
    body: {
        "symbol": str  # 交易对，如 "BTCUSDT"
        "action": str  # "BUY" | "SELL" | "CLOSE"
        "timestamp": str  # ISO 8601 格式
        "indicator_name": str  # TradingView 指标名称
        "confidence": float  # 0.0-1.0
        "price": float  # 建议价格（可选）
        "metadata": dict  # 其他元数据
    }
}
```

#### 输出数据结构
```python
RawSignal {
    signal_id: str  # UUID，系统生成
    received_at: datetime  # 接收时间戳（系统时间）
    source: "tradingview"  # 信号来源
    raw_data: dict  # 原始 Webhook 数据（完整保留）
    signature_valid: bool  # 签名验证结果
}
```

#### 依赖关系
- **依赖**: 无（最外层入口）
- **被依赖**: SignalParser

#### 一致性要求
- **强一致性**: 信号接收必须立即处理，不能丢失
- **同步处理**: 必须同步验证签名，异步传递会引入安全风险

---

### 2.2 SignalParser（信号解析器）

#### 职责（必须做什么）
- 解析 RawSignal，提取标准化字段
- 验证信号格式完整性
- 去重（基于 signal_id 或时间窗口）
- 时间戳对齐和验证
- 将标准化信号传递给 StrategyManager

#### 禁止职责（绝对不允许做什么）
- ❌ 修改信号内容（只能标准化格式）
- ❌ 执行交易决策
- ❌ 缓存信号（必须实时传递）
- ❌ 进行风控检查
- ❌ 连接数据库（由 Repository 层负责）

#### 输入数据结构
```python
RawSignal {
    signal_id: str
    received_at: datetime
    source: str
    raw_data: dict
    signature_valid: bool
}
```

#### 输出数据结构
```python
StandardizedSignal {
    signal_id: str  # 继承自 RawSignal
    symbol: str  # 交易对
    action: "BUY" | "SELL" | "CLOSE"  # 标准化动作
    timestamp: datetime  # 信号生成时间（来自 TradingView）
    received_at: datetime  # 系统接收时间
    indicator_name: str  # 指标名称
    confidence: float  # 0.0-1.0
    price_hint: float | None  # 建议价格（可选）
    metadata: dict  # 保留原始元数据
    is_duplicate: bool  # 是否重复信号
}
```

#### 依赖关系
- **依赖**: SignalReceiver
- **被依赖**: StrategyManager

#### 一致性要求
- **强一致性**: 信号解析必须立即完成，不能延迟
- **同步处理**: 去重检查需要同步，确保不重复处理

---

### 2.3 StrategyManager（策略管理器）⭐ 核心模块

#### 职责（必须做什么）
- 管理所有策略实例（Active/Shadow/Candidate）
- 根据信号类型路由到对应的策略执行器
- 维护策略状态（通过 StrategyStateMachine）
- 协调策略切换流程
- 确保同时只有一个 Active Strategy
- 管理策略生命周期

#### 禁止职责（绝对不允许做什么）
- ❌ 执行策略逻辑（由 StrategyExecutor 负责）
- ❌ 计算指标（由 Evaluator 负责）
- ❌ 直接操作数据库（由 Repository 负责）
- ❌ 执行订单（由 ExecutionEngine 负责）
- ❌ 修改策略配置（由 StrategyVersionManager 负责）

#### 输入数据结构
```python
StandardizedSignal {
    # 见 SignalParser 输出
}

StrategySwitchRequest {
    from_strategy_id: str | None
    to_strategy_id: str
    switch_type: "auto" | "manual"
    reason: str
    requested_by: str  # "system" | "admin"
}
```

#### 输出数据结构
```python
StrategyRouting {
    strategy_id: str
    strategy_type: "active" | "shadow" | "candidate"
    executor_instance: StrategyExecutor | ShadowExecutor
    should_execute: bool  # Active 为 True，Shadow 为模拟
}

StrategyStatus {
    strategy_id: str
    version: str
    status: "candidate" | "shadow" | "active" | "deprecated"
    created_at: datetime
    last_updated: datetime
    metrics_summary: dict  # 最新指标摘要
}
```

#### 依赖关系
- **依赖**: 
  - StrategyVersionManager（获取策略版本信息）
  - StrategyStateMachine（状态转换）
  - StrategyExecutor / ShadowExecutor（执行策略）
  - StrategyRepository（持久化状态）
- **被依赖**: 
  - SignalParser（接收信号）
  - PromotionEngine（触发切换）
  - EliminationEngine（触发切换）

#### 一致性要求
- **强一致性**: 策略状态变更必须立即生效，不能有延迟
- **事务性**: 策略切换必须原子性，不能出现中间状态
- **同步处理**: 状态查询必须同步，确保实时性

---

### 2.4 StrategyVersionManager（策略版本管理器）

#### 职责（必须做什么）
- 管理策略版本元数据
- 版本创建、查询、更新
- 维护版本历史记录
- 计算策略逻辑哈希值（用于验证一致性）
- 管理策略配置快照

#### 禁止职责（绝对不允许做什么）
- ❌ 执行策略逻辑
- ❌ 修改策略状态（由 StrategyStateMachine 负责）
- ❌ 评估策略性能（由 Evaluator 负责）
- ❌ 直接操作策略实例

#### 输入数据结构
```python
StrategyVersionCreate {
    strategy_id: str
    version: str  # "major.minor.patch"
    created_by: str  # "auto_optimizer" | "manual" | "backtest"
    config: dict  # 策略配置（JSON）
    signal_source: str  # TradingView 指标名称
    logic_code: str  # Python 策略逻辑代码
    backtest_metrics: dict  # 回测指标（可选）
}

StrategyVersionQuery {
    strategy_id: str | None
    version: str | None
    status: str | None
}
```

#### 输出数据结构
```python
StrategyVersion {
    version_id: str  # UUID
    strategy_id: str
    version: str
    created_at: datetime
    created_by: str
    config: dict
    signal_source: str
    logic_hash: str  # SHA256 哈希值
    backtest_metrics: dict
    status: "candidate" | "shadow" | "active" | "deprecated"
    promotion_conditions: dict
}
```

#### 依赖关系
- **依赖**: StrategyRepository（持久化）
- **被依赖**: StrategyManager, StrategyExecutor

#### 一致性要求
- **最终一致性**: 版本创建可以异步，但查询必须实时
- **可异步**: 版本历史归档可以异步处理

---

### 2.5 StrategyStateMachine（策略状态机）

#### 职责（必须做什么）
- 管理策略状态转换（Candidate → Shadow → Active → Deprecated）
- 验证状态转换的合法性
- 执行状态转换的副作用（如通知、日志）
- 维护状态转换历史

#### 禁止职责（绝对不允许做什么）
- ❌ 决定何时转换（由 PromotionEngine/EliminationEngine 决定）
- ❌ 执行策略逻辑
- ❌ 直接修改数据库（通过 Repository）

#### 输入数据结构
```python
StateTransitionRequest {
    strategy_id: str
    from_state: "candidate" | "shadow" | "active" | "deprecated"
    to_state: "candidate" | "shadow" | "active" | "deprecated"
    trigger: "auto" | "manual"
    reason: str
    requested_by: str
    conditions_met: dict  # 满足的条件列表
}
```

#### 输出数据结构
```python
StateTransitionResult {
    success: bool
    strategy_id: str
    from_state: str
    to_state: str
    transitioned_at: datetime
    error: str | None
    side_effects: list[str]  # 执行的副作用列表
}
```

#### 依赖关系
- **依赖**: 
  - StrategyRepository（持久化状态）
  - AlertSystem（状态变更通知）
- **被依赖**: StrategyManager, PromotionEngine, EliminationEngine

#### 一致性要求
- **强一致性**: 状态转换必须原子性，不能出现中间状态
- **事务性**: 状态转换必须在一个事务中完成
- **同步处理**: 状态转换必须同步，确保立即生效

---

### 2.6 StrategyExecutor（策略执行器）⭐ 唯一真理版本

#### 职责（必须做什么）
- **执行所有交易决策逻辑**（唯一真理版本）
- 根据信号和当前持仓状态，生成交易决策
- 计算目标仓位、止损止盈位
- 维护策略内部状态（持仓、未完成订单）
- 确保策略逻辑可追溯、可复现

#### 禁止职责（绝对不允许做什么）
- ❌ 直接下单（由 ExecutionEngine 负责）
- ❌ 进行风控检查（由 RiskManager 负责）
- ❌ 计算性能指标（由 Evaluator 负责）
- ❌ 修改策略配置（由 StrategyVersionManager 负责）
- ❌ 连接外部 API（由 Adapter 层负责）

#### 输入数据结构
```python
SignalDecisionInput {
    signal: StandardizedSignal
    current_positions: list[Position]  # 当前持仓
    pending_orders: list[Order]  # 未完成订单
    market_data: MarketData  # 当前市场数据
    strategy_config: dict  # 策略配置
}

Position {
    symbol: str
    side: "long" | "short"
    quantity: float
    entry_price: float
    current_price: float
    unrealized_pnl: float
}

MarketData {
    symbol: str
    bid_price: float
    ask_price: float
    last_price: float
    timestamp: datetime
    order_book: OrderBook  # 订单簿（可选）
}
```

#### 输出数据结构
```python
TradingDecision {
    decision_id: str  # UUID
    strategy_id: str
    signal_id: str
    decision_type: "OPEN" | "CLOSE" | "ADJUST" | "NO_ACTION"
    symbol: str
    side: "BUY" | "SELL" | None
    quantity: float | None
    price_type: "MARKET" | "LIMIT"
    limit_price: float | None
    stop_loss: float | None
    take_profit: float | None
    reason: str  # 决策原因（可追溯）
    confidence: float  # 决策置信度
    timestamp: datetime
    strategy_state_snapshot: dict  # 策略状态快照（用于复现）
}
```

#### 依赖关系
- **依赖**: 
  - StrategyVersionManager（获取策略配置和逻辑）
  - PositionManager（查询持仓）
  - MarketDataAdapter（获取市场数据）
- **被依赖**: StrategyManager

#### 一致性要求
- **强一致性**: 决策生成必须基于最新的持仓和市场数据
- **同步处理**: 决策生成必须同步，确保实时性
- **确定性**: 相同输入必须产生相同输出（可复现）

---

### 2.7 ShadowExecutor（影子策略执行器）

#### 职责（必须做什么）
- 执行与 StrategyExecutor 完全相同的策略逻辑
- 使用真实市场数据进行模拟执行
- 记录模拟交易结果（用于评估）
- 维护独立的模拟账户状态
- 确保与 Active Strategy 使用相同的策略代码

#### 禁止职责（绝对不允许做什么）
- ❌ 真实下单（必须使用 MarketSimulator）
- ❌ 修改真实账户状态
- ❌ 影响 Active Strategy 的执行
- ❌ 使用虚假或历史市场数据（必须用实时数据）

#### 输入数据结构
```python
# 与 StrategyExecutor 相同
SignalDecisionInput {
    # 见 StrategyExecutor
}
```

#### 输出数据结构
```python
ShadowTradingDecision {
    # 与 TradingDecision 相同结构
    # 但增加以下字段：
    is_simulated: True  # 标识为模拟
    simulated_fill_price: float  # 模拟成交价
    simulated_slippage: float  # 模拟滑点
}
```

#### 依赖关系
- **依赖**: 
  - StrategyExecutor（复用策略逻辑）
  - MarketSimulator（模拟成交）
  - MetricsRepository（记录模拟交易）
- **被依赖**: StrategyManager

#### 一致性要求
- **最终一致性**: 模拟交易记录可以异步写入
- **可异步**: 模拟执行可以异步处理，不影响 Active

---

### 2.8 MarketSimulator（市场模拟器）

#### 职责（必须做什么）
- 基于真实订单簿模拟订单成交
- 计算模拟滑点
- 模拟限价单和市价单的成交逻辑
- 返回模拟成交结果

#### 禁止职责（绝对不允许做什么）
- ❌ 真实下单
- ❌ 修改真实市场数据
- ❌ 缓存订单簿（必须使用实时数据）

#### 输入数据结构
```python
SimulatedOrder {
    symbol: str
    side: "BUY" | "SELL"
    quantity: float
    price_type: "MARKET" | "LIMIT"
    limit_price: float | None
    timestamp: datetime
}
```

#### 输出数据结构
```python
SimulatedFill {
    order_id: str  # 模拟订单ID
    symbol: str
    side: "BUY" | "SELL"
    quantity: float
    fill_price: float  # 模拟成交价
    slippage: float  # 滑点（相对于预期价格）
    fill_time: datetime
    order_book_snapshot: dict  # 使用的订单簿快照
}
```

#### 依赖关系
- **依赖**: MarketDataAdapter（获取实时订单簿）
- **被依赖**: ShadowExecutor

#### 一致性要求
- **强一致性**: 必须使用最新的订单簿数据
- **同步处理**: 模拟成交计算必须同步

---

### 2.9 RiskManager（风控管理器）⭐ 强一致性

#### 职责（必须做什么）
- 检查单笔交易风险（仓位、资金、止损）
- 检查账户级风险（总仓位、VaR、集中度）
- 检查策略级风险（连续亏损、回撤）
- 返回风控检查结果和拒绝原因

#### 禁止职责（绝对不允许做什么）
- ❌ 执行订单（由 ExecutionEngine 负责）
- ❌ 修改订单参数（只能批准或拒绝）
- ❌ 计算性能指标（由 Evaluator 负责）
- ❌ 绕过风控检查（所有交易必须通过）

#### 输入数据结构
```python
RiskCheckRequest {
    decision: TradingDecision
    current_positions: list[Position]
    account_balance: float
    account_equity: float
    strategy_metrics: dict  # 策略最新指标
    risk_config: dict  # 风控配置
}
```

#### 输出数据结构
```python
RiskCheckResult {
    passed: bool
    decision_id: str
    rejected_reasons: list[str]  # 如果 rejected
    risk_level: "LOW" | "MEDIUM" | "HIGH"
    risk_metrics: {
        position_risk: float  # 仓位风险
        account_risk: float  # 账户风险
        var: float  # 风险价值
        concentration: float  # 集中度
    }
    recommendations: list[str]  # 建议（如降低仓位）
}
```

#### 依赖关系
- **依赖**: 
  - PositionManager（查询持仓）
  - AccountManager（查询账户信息）
  - MetricsRepository（查询策略指标）
- **被依赖**: StrategyExecutor, ExecutionEngine

#### 一致性要求
- **强一致性**: 风控检查必须基于最新数据，不能有延迟
- **同步处理**: 必须同步检查，不能异步
- **不可绕过**: 所有交易必须通过风控检查

---

- 计算持仓盈亏
- 维护持仓状态（开仓、平仓、调整）
- 提供持仓聚合查询（按策略、按交易对）

#### 禁止职责（绝对不允许做什么）
- ❌ 直接修改持仓（由 ExecutionEngine 负责）
- ❌ 执行订单
- ❌ 进行风控检查

#### 输入数据结构
```python
PositionQuery {
    strategy_id: str | None
    symbol: str | None
    include_closed: bool  # 是否包含已平仓
}
```

#### 输出数据结构
```python
Position {
    position_id: str
    strategy_id: str
    symbol: str
    side: "long" | "short"
    quantity: float
    entry_price: float
    current_price: float
    unrealized_pnl: float
    realized_pnl: float
    opened_at: datetime
    last_updated: datetime
}
```

#### 依赖关系
- **依赖**: TradeRepository（查询交易记录）
- **被依赖**: StrategyExecutor, RiskManager, ExecutionEngine

#### 一致性要求
- **强一致性**: 持仓查询必须实时，基于最新交易记录
- **同步处理**: 持仓查询必须同步

---

### 2.11 AccountManager（账户管理器）

#### 职责（必须做什么）
- 查询账户余额和权益
- 计算可用资金
- 维护账户状态快照
- 提供账户级统计信息

#### 禁止职责（绝对不允许做什么）
- ❌ 直接修改账户余额（由交易所 API 负责）
- ❌ 执行交易
- ❌ 进行风控决策

#### 输入数据结构
```python
AccountQuery {
    include_history: bool  # 是否包含历史快照
    snapshot_date: datetime | None  # 查询特定日期的快照
}
```

#### 输出数据结构
```python
AccountInfo {
    account_id: str
```

## 三、一致性要求分类

### 3.1 强一致性模块（必须同步、立即生效）

这些模块的操作必须立即完成，不能有延迟，且必须基于最新数据：

1. **SignalReceiver** - 信号接收不能丢失
2. **SignalParser** - 信号解析必须立即完成
3. **StrategyManager** - 策略状态查询和切换必须实时
4. **StrategyStateMachine** - 状态转换必须原子性
5. **StrategyExecutor** - 决策生成必须基于最新数据
6. **RiskManager** - 风控检查必须同步，不能绕过
7. **PositionManager** - 持仓查询必须实时
8. **AccountManager** - 账户信息查询必须实时
9. **ExecutionEngine** - 订单执行必须立即提交
10. **EliminationEngine** - 严重风险事件的淘汰检查必须立即进行
11. **StrategyRepository** - 策略状态更新必须立即持久化
12. **TradeRepository** - 交易记录必须立即持久化
13. **TradingViewAdapter** - Webhook 接收必须立即处理
14. **ExchangeAdapter** - 订单操作必须同步
15. **MarketDataAdapter** - 价格查询必须同步

### 3.2 最终一致性模块（可以异步、允许延迟）

这些模块的操作可以异步进行，允许一定延迟：

1. **StrategyVersionManager** - 版本创建可以异步
2. **ShadowExecutor** - 模拟执行可以异步
3. **MarketSimulator** - 模拟成交计算可以异步
4. **OrderManager** - 订单状态同步可以异步
5. **Evaluator** - 评估任务可以异步执行
6. **MetricsCalculator** - 指标计算可以异步
7. **PromotionEngine** - 晋升检查可以定期异步进行
8. **Optimizer** - 优化计算可以异步执行
9. **SystemMonitor** - 监控数据可以异步收集
10. **HealthChecker** - 健康检查和恢复可以异步
11. **AlertSystem** - 告警通知可以异步发送
12. **MetricsRepository** - 指标存储可以异步
13. **LogRepository** - 日志存储可以异步

## 四、唯一状态真理源（Single Source of Truth）

### 4.1 策略状态真理源

**位置**: `StrategyRepository` + `StrategyStateMachine`

**说明**:
- 所有策略的状态（Candidate/Shadow/Active/Deprecated）存储在 `StrategyRepository`
- 状态转换由 `StrategyStateMachine` 统一管理
- 其他模块查询策略状态必须通过 `StrategyRepository`，不能缓存

### 4.2 持仓状态真理源

**位置**: `TradeRepository` + `PositionManager`

**说明**:
- 所有交易记录存储在 `TradeRepository`
- `PositionManager` 基于交易记录计算当前持仓
- 持仓状态不单独存储，而是通过交易记录实时计算
- 交易所 API 的持仓信息作为验证参考，但不作为真理源

### 4.3 账户状态真理源

**位置**: `ExchangeAdapter`（交易所 API）

**说明**:
- 账户余额、权益等状态以交易所 API 为准
- `AccountManager` 负责查询和缓存，但真理源是交易所
- 系统内部计算的账户状态仅用于风控和决策，不作为最终真理

### 4.4 策略逻辑真理源

**位置**: `StrategyExecutor`（Python 代码）

**说明**:
- 所有交易决策逻辑在 `StrategyExecutor` 中实现
- TradingView 的 Pine Script 仅提供信号，不包含交易逻辑
- 策略逻辑代码通过 `StrategyVersionManager` 管理版本
- 策略逻辑的哈希值存储在版本元数据中，用于验证一致性

### 4.5 市场数据真理源

**位置**: `MarketDataAdapter`（交易所市场数据 API）

**说明**:
- 实时价格、订单簿等市场数据以交易所 API 为准
- 可以缓存以减少 API 调用，但必须定期更新
- Shadow Strategy 必须使用相同的市场数据源

### 4.6 交易记录真理源

**位置**: `TradeRepository`

**说明**:
- 所有交易记录（Active + Shadow）存储在 `TradeRepository`
- 交易记录一旦写入，不可修改（除非数据修复）
- 交易所的交易记录作为验证参考，但系统内部的 `TradeRepository` 是真理源

## 六、关键设计原则

1. **单一职责**: 每个模块只负责一个明确的职责
2. **依赖倒置**: 高层模块不依赖低层模块，都依赖抽象接口
3. **数据流向**: 信号 → 决策 → 执行 → 反馈，单向流动
4. **真理源唯一**: 每个状态只有一个真理源，其他都是缓存或计算
5. **强一致性优先**: 涉及资金和风险的操作必须强一致性
6. **异步优化**: 非关键路径可以异步处理，提高性能
7. **可追溯性**: 所有操作都有日志，可以追溯
8. **可测试性**: 模块之间通过接口交互，便于单元测试
