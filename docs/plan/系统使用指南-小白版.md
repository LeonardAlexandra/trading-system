# 策略优化进化闭环系统 - 使用指南（小白版）

## ⚠️ Phase1.0 范围与限制（Scope & Limits）

**重要提示**：本文档描述的是 Phase1.0 版本的功能和限制。在使用系统前，请务必了解以下限制：

### Phase1.0 已知限制

1. **Paper 模式限制**:
   - Phase1.0 仅支持 Paper 模式（模拟交易，下单即成交）
   - 不支持真实交易所异步成交
   - 不支持订单部分成交
   - Paper 模式下所有订单立即成交，不模拟真实市场延迟

2. **单实例限制**:
   - Phase1.0 仅支持单实例运行（`workers=1`）
   - 不支持多进程/多实例部署
   - 不支持水平扩展
   - 必须使用 `uvicorn --workers 1` 启动

3. **单交易所/单产品限制**:
   - Phase1.0 固定支持 1 家交易所 + 1 种产品形态（spot/perp 二选一）
   - 不支持多交易所/多产品形态
   - 切换交易所或产品形态需要修改配置并重启系统

4. **无消息队列**:
   - Phase1.0 不使用 Celery/Redis/消息队列
   - 定时任务使用进程内调度（ExecutionWorker）
   - 不支持分布式任务调度

5. **功能限制**:
   - ❌ 不支持多策略并发运行（只能运行 1 个策略）
   - ❌ 不支持策略自动切换
   - ❌ 不支持 Shadow Strategy（影子策略测试）
   - ❌ 不支持策略性能自动评估和优化
   - ❌ 不支持复杂的监控仪表板
   - ❌ 不支持短信/电话告警（仅支持文件日志）
   - ❌ 不支持订单改价功能（仅支持查询、取消、状态同步）
   - ❌ 不支持自动对账功能（持仓管理基于成交驱动更新）

### Phase1.0 实际实现说明

以下功能在 Phase1.0 中的实际实现情况：

- **账户管理**: 支持查询账户信息（通过 ExchangeAdapter），但不支持账户余额自动同步
- **订单取消与同步**: 支持订单查询、取消、状态同步，但不支持订单改价
- **持仓管理**: 基于成交驱动更新（position_snapshot 投影表），不支持自动对账
- **日志系统**: 文件日志为主，关键事件写入 execution_events 表，不支持全量数据库日志

**注意**: 以上限制将在后续版本（Phase 2.0+）中逐步解除。Phase1.0 的目标是构建一个**安全、可扩展、不会推翻重来**的实盘基础系统。

## 📖 这是什么系统？

这是一个**自动交易系统**，它可以：
- 🤖 **自动接收交易信号**：从 TradingView 接收买卖信号
- 💰 **自动执行交易**：根据信号自动在交易所下单
- 🛡️ **自动风控保护**：确保每笔交易都经过安全检查
- 📊 **自动记录和分析**：记录所有交易，方便后续分析
- 🔄 **长期自动运行**：可以 7x24 小时无人值守运行

## 🎯 系统能实现什么效果？

### 1. 自动化交易流程

```
TradingView 发出信号
    ↓
系统自动接收信号
    ↓
系统分析并生成交易决策
    ↓
系统进行风控检查
    ↓
系统自动下单到交易所
    ↓
系统记录交易结果
```

**你只需要：**
- ✅ 在 TradingView 上设置好指标
- ✅ 配置好系统参数
- ✅ 启动系统
- ✅ 定期查看运行状态

**系统会自动完成：**
- ✅ 接收信号
- ✅ 分析决策
- ✅ 风控检查
- ✅ 执行交易
- ✅ 记录日志

### 2. 安全保障机制

#### 🛡️ 多层风控保护

1. **单笔交易风控**
   - 每笔交易都有最大仓位限制
   - 每笔交易都有止损保护
   - 防止单笔交易亏损过大

2. **账户级风控**
   - 限制总仓位大小
   - 限制单日最大亏损
   - 限制账户最大回撤

3. **策略级风控**
   - 监控连续亏损
   - 监控策略表现
   - 表现不佳时自动降级或停止

#### 🔒 安全特性

- ✅ **信号验证**：验证 TradingView 信号的真实性
- ✅ **订单幂等**：防止重复下单
- ✅ **持仓一致性**：自动对账，确保持仓准确
- ✅ **异常恢复**：网络中断、API 错误时自动恢复
- ✅ **完整审计**：所有操作都有日志记录

### 3. 可追溯性

每笔交易都可以完整追溯：

```
信号 ID → 决策 ID → 执行 ID → 交易 ID
```

你可以查询：
- 📝 为什么做出这个交易决策？
- 🛡️ 风控检查的结果是什么？
- 💰 订单执行的情况如何？
- 📊 最终的交易结果是什么？

### 4. 系统状态管理

- ✅ **策略配置管理**：可以保存和切换不同的策略配置
- ✅ **持仓管理**：自动跟踪和管理持仓
- ✅ **账户管理**：实时查询账户余额和权益
- ✅ **订单管理**：查询、取消、同步订单状态

## 🚀 如何使用这个系统？

### 第一步：准备工作

#### 1.1 准备 TradingView 信号源

1. **在 TradingView 上创建指标**
   - 使用 Pine Script 编写你的交易策略
   - 指标需要能够发出 BUY/SELL/CLOSE 信号
   - 设置 Webhook 发送信号

2. **获取 Webhook 密钥**
   - 在系统配置中设置 `TV_WEBHOOK_SECRET`
   - 用于验证 TradingView 信号的真实性

#### 1.2 准备交易所账户

1. **选择交易所**
   - 系统支持币安（Binance）、OKX、Bybit 等
   - MVP 版本固定支持 1 家交易所

2. **创建 API Key**
   - 在交易所创建 API Key 和 Secret
   - 设置适当的权限（交易、查询）
   - **重要**：建议先使用 Paper Trading（模拟交易）测试

3. **配置 API 信息**
   - 在系统配置中设置 `EXCHANGE_API_KEY`
   - 在系统配置中设置 `EXCHANGE_API_SECRET`

#### 1.3 准备数据库

- **开发环境**：可以使用 SQLite（简单）
- **生产环境**：建议使用 PostgreSQL（稳定）

---

### 第二步：配置系统

#### 2.1 配置文件示例

创建配置文件 `config/production.yaml`：

```yaml
# TradingView 配置
tradingview:
  webhook_secret: "your_webhook_secret_here"  # 从环境变量读取
  
# 交易所配置
exchange:
  name: "binance"  # 固定 1 家交易所
  api_key: "${EXCHANGE_API_KEY}"  # 从环境变量读取
  api_secret: "${EXCHANGE_API_SECRET}"  # 从环境变量读取
  sandbox: true  # true=模拟交易, false=真实交易（建议先用 true 测试）
  
# 产品类型
product_type: "spot"  # "spot" 或 "perp"（固定 1 种）

# 策略配置
strategy:
  strategy_id: "SMC_FVG_v1"  # 策略名称
  config:
    max_position_size: 0.1  # 最大仓位（BTC）
    stop_loss_pct: 0.02      # 止损 2%
    
# 风控配置
risk:
  max_single_trade_risk: 0.01  # 单笔最大风险 1%
  max_account_risk: 0.05       # 账户最大风险 5%
  max_drawdown: 0.20            # 最大回撤 20%
  
# 数据库配置
database:
  url: "${DATABASE_URL}"  # 从环境变量读取
  
# 日志配置
logging:
  level: "INFO"
  file: "/var/log/trading_system/app.log"
  database: true
```

#### 2.2 环境变量设置

创建 `.env` 文件：

```bash
# TradingView
TV_WEBHOOK_SECRET=your_webhook_secret

# 交易所
EXCHANGE_API_KEY=your_api_key
EXCHANGE_API_SECRET=your_api_secret

# 数据库
DATABASE_URL=postgresql://user:password@localhost:5432/trading_system
```

### 处理异常

#### 如果系统停止运行

1. **检查日志**
   ```bash
   tail -100 /var/log/trading_system/app.log
   ```

2. **检查系统状态**
   ```bash
   systemctl status trading-system
   ```

3. **重启系统**
   ```bash
   systemctl restart trading-system
   ```

#### 如果收到告警

1. **查看告警详情**
   - 检查邮件告警内容
   - 查看系统日志中的告警记录

2. **根据告警级别处理**
   - **CRITICAL（紧急）**：立即检查，可能需要停止交易
   - **IMPORTANT（重要）**：及时处理，可能需要人工干预
   - **WARNING（警告）**：关注趋势，可能需要调整参数
   - **INFO（信息）**：了解即可，正常状态更新

#### 如果持仓不一致

系统会自动处理：
- **轻微不一致（< 1%）**：仅记录日志，不影响运行
- **中等不一致（1-5%）**：发送告警，不覆盖持仓
- **严重不一致（> 5%）**：自动覆盖持仓，进入安全模式，暂停新订单

## 🔍 常见问题

### Q1: 系统如何保证不会重复下单？

**A:** 系统使用 `decision_id` 作为订单的唯一标识：
- 每个交易决策都有唯一的 `decision_id`
- 系统在数据库中记录 `decision_id → order_id` 映射
- 如果相同的 `decision_id` 再次提交，系统会返回已存在的订单，不会重复下单

### Q2: 如果网络中断怎么办？

**A:** 系统有自动恢复机制：
- 检测到网络中断时，会记录错误日志
- 自动重试机制（最多 3 次）
- 网络恢复后自动继续运行
- 中断期间的信号可能会丢失，但不会影响系统稳定性

### Q3: 如何查看某笔交易的完整流程？

**A:** 系统记录了完整的交易链路：
```
signal_id → decision_id → execution_id → trade_id
```

可以通过以下方式查询：
- 通过 `signal_id` 查询：查看信号详情
- 通过 `decision_id` 查询：查看决策详情和原因
- 通过 `execution_id` 查询：查看订单执行详情
- 通过 `trade_id` 查询：查看最终交易结果

### Q4: 系统可以同时运行多个策略吗？

**A:** MVP 版本不支持多策略并发运行，只能运行 1 个策略。多策略支持将在后续版本中添加。

### Q5: 如何修改策略参数？

**A:** 
1. 修改配置文件中的策略参数
2. 重启系统（或通过 API 热更新，如果支持）
3. 系统会加载新的配置

### Q6: 系统如何保证持仓准确？

**A:** Phase1.0 使用成交驱动更新机制：
1. **成交驱动更新**：订单成交时立即更新持仓（position_snapshot 投影表）
2. **持仓查询**：从 position_snapshot 表读取，不直接查询交易所

**注意**: Phase1.0 不支持自动对账功能。持仓管理基于成交记录实时计算，确保持仓数据的一致性。自动对账功能将在后续版本中添加。
