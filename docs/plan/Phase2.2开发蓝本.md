# Phase 2.2 完整智能 BI 开发交付包（全文版）

**版本**: v2.0.0（全文扩写）  
**创建日期**: 2026-02-07  
**基于**: Phase划分与实现顺序-需求清单.md v2.0.0、Phase2.0开发交付包.md（全文版）、Phase2.1开发交付包.md（全文版）、Phase1.2开发交付包.md（全文版）

**定位**：Phase 2.2 是**纯只读展示层（Read-Only Presentation Layer）**。本文档为系统观察与解释层的边界文档：运维/交易/研发可通过 BI 看清系统发生了什么、追溯为什么会发生、判断系统是否健康运行；**任何 BI 使用者无法通过 BI 改变系统行为**；BI 展示的数据与 Phase 2.0/2.1 完全一致；BI 永远不成为“隐性决策系统”。**禁止**：在 BI 层执行交易、写回参数、触发评估/学习、调用发布/回滚/门禁；新增影响系统行为的逻辑（新口径、好坏判断、第二套指标）；绕过既有 API 直连原始表做私有计算；BI 成为控制面（“在 BI 上点一下就生效”）。

## A. 概述

### A.1 目标

**可审计、可运营**：完整交易统计、资金曲线、决策过程（信号→理由→风控→执行）与学习/变更历史（策略/参数版本、评估与优化历史、门禁/回滚/自动停用）在 BI 上可查可展示。Phase 2.2 为**纯只读展示层**：不执行交易、不写回参数、不触发评估/学习、不调用发布/回滚/门禁；所有数据来自 Phase 1.2/2.0/2.1 的既有契约，与 2.0/2.1 完全一致；职责仅限于**展示、对齐、解释、审计、运营观察**。

### A.2 完成判定

以下条件**全部**满足时，Phase 2.2 视为完成；**任一条未达成则禁止进入后续 Phase 或对外展示**。

- （1）TBI-1、TBI-2 验收通过；E 节每项具备输入/输出/可验证断言且已执行验收。
- （2）**只读边界**：BI 层未写入任何业务表（1.2/2.0/2.1）；未调用任何会改变状态的接口（评估、学习、发布、回滚、门禁、写回）；未持有执行权限；验收可验证（无写操作、无状态变更调用）。
- （3）**指标与统计完全复用既有口径**：所有统计指标来自 Phase 2.0 MetricsRepository / evaluation_report 或通过 2.0 只读 API 获取；BI 层未重新定义计算公式、未派生“第二套指标”；资金曲线/权益曲线数据来源与时间粒度、聚合方式已在文档写死并与 2.0 一致。
- （4）**决策过程为“解释层”**：仅展示 Phase 1.2 的决策输入快照与追溯结果；BI 未在展示层生成“新解释”或“推断原因”；PARTIAL/NOT_FOUND 追溯结果清晰展示缺失原因与 missing_nodes。
- （5）**学习/版本/发布历史与 2.1 完全对齐**：参数版本、策略版本、评估历史、发布门禁、回滚、自动停用展示全部来自 Phase 2.1 的 release_audit/learning_audit（或 2.1 只读 API）；展示字段、时间顺序、状态含义与 2.1 定义一致；BI 未合成“综合评分”“自动结论”。
- （6）**数据一致性**：BI 展示的指标、评估结论、版本与审计与 2.0/2.1 API 查询结果一致；验收可抽检比对。
- （7）**权限与脱敏**：最小权限模型（viewer/operator/admin）与脱敏字段已在交付包中写死；若多租户不在本 Phase 实现，已明确“当前单租户/内网使用”。

### A.3 Phase 2.2 终止条件与禁止进入后续/对外展示的情形（写死）

- **视为完成**：A.2 全部达成且 F 节端到端用例通过。
- **禁止进入后续 Phase 或对外正式展示**的情形（任一条即禁止）：
  - BI 展示数据与 2.0/2.1 不一致（口径分叉、自行计算或推断结论）。
  - BI 能触发系统状态变化（如调用发布/回滚/门禁/评估/学习/写回接口，或“在 BI 上点一下就生效”）。
  - BI 自行计算或推断交易/学习结论（第二套指标、综合评分、自动好坏判断）。
  - BI 绕过既有 API 直连原始表做私有计算。
  - 决策过程展示在 BI 层生成“新解释”或“应该怎么做”类推断。

### A.4 前置依赖

- Phase 2.0/2.1 的指标、评估、策略版本与发布审计可用（evaluation_report、metrics_snapshot、release_audit、learning_audit 可查）；Phase 1.2 的追溯 API、决策输入快照可查。

### A.5 In-Scope / Out-of-Scope（写死）

| 类别 | 内容 |
|------|------|
| **In-Scope** | 按日/周/策略的完整交易统计与资金曲线 **只读 API**（数据来自 2.0）；决策过程展示 **只读 API**（数据来自 1.2 追溯与决策快照）；策略/参数版本变更、评估报告、门禁/回滚/自动停用历史 **只读 API**（数据来自 2.0/2.1）；至少一个 BI 前端页面或模块集成上述能力；与 2.0/2.1 的数据契约与**只读**复用；权限与脱敏最小约定；可选 BI 层缓存/物化视图（仅性能优化、语义与 2.0 一致）。 |
| **Out-of-Scope** | 策略或参数写回、Evaluator/Optimizer 执行、发布门禁与回滚**触发**（属 2.1）；在 BI 层新增计算口径、好坏判断、第二套指标；BI 成为控制面；多租户完整实现若不在本包则写死为单租户/内网。 |

## B. 架构级硬约束（0.1～0.4 与只读边界）

**Phase 2.2 的“只读边界”**为全文主轴，必须反复、一致地体现：BI 层**不写任何业务表**、**不调用任何会改变状态的接口**、**不持有执行权限**、**不具备发布/回滚/学习/评估的触发能力**。上述约束必须体现在 B、A.2、E、F 中。

### B.0 Phase 2.2 只读边界（写死）

- **BI 层不写任何业务表**：不对 Phase 1.2/2.0/2.1 的 trade、decision_snapshot、execution、metrics_snapshot、evaluation_report、release_audit、learning_audit、strategy_version、param_version 等执行 INSERT/UPDATE/DELETE。若存在 BI 层缓存或物化视图，**仅**用于查询性能优化，且不改变上游业务语义；缓存更新策略与失效条件在交付包中写死。
- **BI 层不调用任何会改变状态的接口**：不调用 Evaluator.evaluate、Optimizer.suggest 的写回路径、ReleaseGate.submit_candidate/confirm_manual/apply_approved/rollback_to_stable、MetricsRepository.write（或任何写入 2.0/2.1 表的接口）。仅允许调用**只读**查询类 API（如按条件查 trade、查 evaluation_report、查 release_audit、查追溯、查决策快照）。
- **BI 层不持有执行权限**：不暴露“执行交易”“提交订单”“触发策略”等能力；不持有可写 Token 或写权限用于 1.2/2.0/2.1 业务表。
- **BI 层不具备发布/回滚/学习/评估的触发能力**：页面或接口中**不能**出现“在 BI 上点一下就生效”的按钮或调用（如“通过门禁”“执行回滚”“触发评估”“应用参数”）；仅可展示门禁/回滚/评估的**历史结果**，不可触发。

| 约束 | 在本 Phase 的落地 |
|------|------------------|
| **0.1 学习边界** | 不实现。BI 仅**展示**参数/版本变更与评估结果，不执行任何参数写回或学习逻辑，不触发 Optimizer。 |
| **0.2 Evaluator Contract** | **只读依赖**。展示的评估报告内容与 2.0 持久化 schema 一致（目标函数、约束、基线、结论等）；**不产生新评估**、不调用 Evaluator.evaluate。 |
| **0.3 发布门禁与回滚** | **只读依赖**。展示门禁/回滚/自动停用**历史**（来自 2.1 release_audit 或只读 API）；**不执行**门禁或回滚，不调用 ReleaseGate 的写或状态变更接口。 |
| **0.4 决策输入快照** | **只读依赖**。决策过程展示**仅**引用 1.2 决策输入快照与追溯 API（按 decision_id/signal_id 或时间范围）；不写入快照，不在 BI 层生成新解释或推断。 |

### B.1 禁止 BI 层新增影响系统行为的逻辑（写死）

- **不得新增计算口径**：所有指标与统计**必须**复用 Phase 2.0 的 MetricsRepository / evaluation_report 口径；不得在 BI 层重新定义“盈亏”“胜率”“回撤”等计算公式。
- **不得新增“判断好坏”的业务规则**：不在 BI 层产出“建议调整”“综合评分”“自动结论”等；仅展示 2.0 的 conclusion、2.1 的 action 等既有字段。
- **不得在 BI 层派生“第二套指标”**：不基于 trade/execution 在 BI 侧再算一套“近似”指标；若需聚合，**仅**通过调用 2.0 只读 API 或读取 2.0 已持久化的 metrics_snapshot/evaluation_report。

## C. 数据模型与 Schema

本 Phase **不新增业务写库表**；所有展示数据**必须**来自 Phase 1.2/2.0/2.1 的既有契约与只读 API，**不得**绕过既有 API 直连原始表做“私有计算”。

### C.1 依赖数据源（只读，写死）

| 数据类别 | 来源 | 用途 | 访问方式 |
|----------|------|------|----------|
| 交易与决策 | Phase 1.2 | 决策过程展示、追溯、决策快照 | 仅通过 1.2 追溯 API、决策快照查询 API；**禁止**直连 trade/decision 表做 BI 自有聚合 |
| 指标与评估 | Phase 2.0 | 统计、资金曲线、评估报告展示 | 仅通过 2.0 MetricsRepository 只读查询、evaluation_report 只读查询；**禁止**在 BI 层重算指标 |
| 策略版本与发布 | Phase 2.0/2.1 | 版本历史、门禁/回滚/自动停用历史 | 仅通过 2.0/2.1 只读 API（strategy_version、param_version、release_audit、learning_audit）；**禁止**写或触发 2.1 状态变更 |

- **资金曲线 / 权益曲线**数据来源（写死）：来自 **trade** 或 **account snapshot** 或 **evaluation_report / metrics_snapshot** 中已聚合的字段；时间粒度与聚合方式（如按日/周、sum/累计）**必须**与 2.0 约定一致，并在本文档或接口说明中写明；**禁止**在 BI 层用自有公式重算“近似”权益曲线。

### C.2 可选 BI 层缓存 / 物化视图（仅性能优化）

- 若引入 BI 层缓存或物化视图（如按日/周/策略的统计快照、权益曲线快照）：
  - **仅**作为性能优化，**不**改变业务语义。
  - **语义必须与 2.0 指标完全一致**：缓存内容来自 2.0 的 metrics_snapshot 或 2.0 只读 API 的查询结果，或按 2.0 同一口径计算后仅做冗余存储；**禁止**在缓存层使用与 2.0 不同的公式或口径。
  - **更新策略与失效条件**必须在交付包中写明（如：按 2.0 评估周期刷新、或 TTL 失效后回源 2.0 查询）；验收时须可验证：关闭缓存或失效后，BI 展示与直接查 2.0 一致。

## D. 接口与边界

**原则**：所有 BI 接口为**只读**（GET 或等价只读查询）；**禁止**提供会触发状态变更的 POST/PUT/DELETE（如“触发评估”“执行回滚”“通过门禁”）。数据来源与 2.0/2.1 对齐；接口说明中显式标注**本 API 为只读，不改变任何业务状态**。

### D.1 完整交易统计与资金曲线（BI-1）

- **数据来源**：Phase 2.0 MetricsRepository、metrics_snapshot、或 2.0 提供的只读聚合 API；**禁止**在 BI 层从 trade 表重新聚合计算。
- **口径**：与 Phase 2.0 B.2 指标口径一致（trade_count、win_rate、realized_pnl、max_drawdown、avg_holding_time 等）；时间粒度与聚合方式与 2.0 约定一致。

```text
GET /api/bi/stats?from=&to=&group_by=day|week|strategy_id
# 返回：盈亏、胜率、最大回撤、交易次数等；字段与 2.0 metrics_snapshot / evaluation_report 对齐
# 本 API 为只读，不改变任何业务状态。数据来源：2.0 只读 API。

GET /api/bi/equity_curve?strategy_id=&from=&to=&granularity=day|week
# 返回：账户或策略维度的权益曲线；数据来源：2.0 或 1.2 约定（trade/account snapshot/metrics）；粒度与 2.0 一致
# 本 API 为只读，不改变任何业务状态。
```

- 多维度筛选与导出：仅增加筛选参数与导出格式，**不**改变统计口径；导出内容与接口返回一致。

### D.2 决策过程与学习/变更展示（BI-2）

- **决策过程**：**只展示** Phase 1.2 的决策输入快照与追溯结果（signal→decision→决策快照→execution→trade）；**不在 BI 层生成“新解释”或“推断原因”**；展示顺序、字段含义、缺失数据的展示方式在交付包中写死；对 **PARTIAL** / **NOT_FOUND** 追溯结果，**必须**清晰展示 trace_status、missing_nodes、missing_reason（或等价说明），不得静默隐藏。
- **学习/版本/发布历史**：全部来自 Phase 2.1 release_audit、learning_audit 及 2.0 evaluation_report、strategy_version/param_version 只读查询；展示字段、时间顺序、状态含义与 2.1 定义一致；**BI 不得合成“综合评分”“自动结论”**；**BI 不得成为“第二个 Release Gate”**（仅展示，不触发）。

```text
GET /api/bi/decision_flow?decision_id=|signal_id=
# 返回：单笔 信号→理由→风控→执行；来自 1.2 追溯 API；含 trace_status、missing_nodes（若 PARTIAL/NOT_FOUND）
# 本 API 为只读，不改变任何业务状态。

GET /api/bi/decision_flow/list?from=&to=&strategy_id=&limit=
# 列表；每条可含 trace_status、missing_nodes
# 本 API 为只读，不改变任何业务状态。

GET /api/bi/version_history?strategy_id=&limit=
# 策略/参数版本变更历史；来自 2.0/2.1 只读 API
# 本 API 为只读，不改变任何业务状态。

GET /api/bi/evaluation_history?strategy_id=&from=&to=
# 评估报告历史；来自 2.0 evaluation_report 只读查询；字段与 2.0 schema 一致
# 本 API 为只读，不改变任何业务状态。

GET /api/bi/release_audit?strategy_id=&from=&to=
# 门禁/回滚/自动停用历史；来自 2.1 release_audit 只读查询；action/gate_type 等与 2.1 定义一致
# 本 API 为只读，不改变任何业务状态。
```

- 错误码：404 未找到、400 参数错误、503 上游不可用；**敏感信息脱敏**在 D.4 与 E 中约定。

### D.3 前端与权限

- 至少一个 BI 页面或模块集成上述只读 API；**禁止**在页面上提供“触发评估”“执行回滚”“通过门禁”“应用参数”等会改变系统状态的按钮或操作。
- 权限与访问控制在 D.4 约定。

### D.4 权限、脱敏与多租户（最小但明确，写死）

- **最小权限模型**（实现时至少支持以下之一或子集，具体在交付包中写死）：
  - **viewer**：只读；可查看统计、曲线、决策过程、版本与评估与审计历史；不可导出敏感字段（若导出则脱敏）。
  - **operator**：在 viewer 基础上可查看运维/审计详情（如完整 release_audit payload、错误日志入口）；仍**不可**触发任何状态变更。
  - **admin**：可查看全部展示数据与配置；仍**不可**通过 BI 触发发布/回滚/评估/学习。
- **必须脱敏的字段**（写死）：API key、交易所密钥、账户标识（若对外）、内部 IP/主机名（若需）；具体字段列表在交付包中列明，验收时检查脱敏生效。
- **多租户**：若本 Phase 不实现多租户，**必须**明确“当前单租户/内网使用”，禁止模糊表述；若实现则约定租户隔离粒度与数据范围。

## E. 任务拆分

每项任务均给出**输入（查询条件）**、**输出（字段与含义）**、**可验证断言**；验收须覆盖：**数据与 2.0/2.1 一致**、**只读性（无状态变更）**、**权限与脱敏**、**大数据量下的性能边界**（若适用）。

| 任务编号 | 目的 | 输入（查询条件） | 输出（字段/含义） | 实现要点 | 可验证断言（验收 checkbox） | 交付物 |

## F. 测试与验收

### F.1 端到端用例

- **E2E-BI（主流程）**  
  - 步骤：打开 BI 页面 → 选择时间范围与策略 → 展示完整统计与资金曲线、决策过程列表、版本与评估与门禁/回滚历史。  
  - 验证：展示数据与 2.0/2.1 同条件 API 查询结果**一致**；**无**“触发评估”“执行回滚”“通过门禁”等可点击并生效的操作。

- **E2E-BI-只读**  
  - 步骤：对 BI 所有入口（页面与 API）执行仅读操作（GET）；监控或审计 1.2/2.0/2.1 业务表或写接口。  
  - 验证：**无**任何 INSERT/UPDATE/DELETE 到 1.2/2.0/2.1 业务表；**无**调用 Evaluator.evaluate、ReleaseGate 的写或状态变更、MetricsRepository.write 等。

- **E2E-BI-一致性**  
  - 步骤：同一时间范围与策略，分别从 BI API 与 2.0/2.1 只读 API 获取统计、评估、审计。  
  - 验证：指标值、评估结论、审计记录**一致**（或 BI 明确标注数据来源与刷新时机，且与 2.0/2.1 一致）。

- **E2E-BI-决策过程与缺失展示**  
  - 步骤：查询一条 PARTIAL 或 NOT_FOUND 的追溯结果。  
  - 验证：页面或 API 返回中**清晰展示** trace_status、missing_nodes（及缺失原因）；**未**在 BI 层生成“新解释”或“应该怎么做”。

### F.2 回归清单

- Phase 2.0 评估查询、Phase 2.1 门禁/回滚与 Phase 1.2 追溯**无回归**；BI 仅为只读消费，未新增对上游的写或触发依赖。
- 指标口径与 2.0 一致，无 BI 侧口径分叉。
- 权限与脱敏按 D.4 生效。

### F.3 禁止进入后续 Phase 或对外正式展示的失败情形（写死）

以下任一条成立则**禁止**进入后续 Phase 或对外正式展示，须在 Phase 2.2 内修复并重新验收：

- BI 展示数据与 2.0/2.1 不一致（口径分叉、自行计算或推断结论）。
- BI 能触发系统状态变化（存在可用的“在 BI 上点一下就生效”的按钮或接口调用）。
- BI 自行计算或推断交易/学习结论（第二套指标、综合评分、自动好坏判断）。
- BI 绕过既有 API 直连原始表做私有计算。
- 决策过程展示在 BI 层生成“新解释”或“应该怎么做”类推断；或 PARTIAL/NOT_FOUND 未清晰展示缺失原因。
- 敏感字段未脱敏或权限模型未按约定生效。

## G. 风险与非功能性要求

- **性能**：大数据量下统计与曲线查询的聚合与缓存策略在交付包中约定；若使用 BI 层缓存/物化视图，**语义与 2.0 完全一致**，更新策略与失效条件写死；验收可验证关闭缓存后与 2.0 一致。
- **权限与脱敏**：最小权限模型（viewer/operator/admin）与必须脱敏字段在 D.4 列明；验收须覆盖权限与脱敏。
- **多租户**：若本 Phase 不实现，**必须**明确“当前单租户/内网使用”，禁止模糊表述。
- **实现选择（写死）**：若存在多种实现方式，采用**最保守、最清晰、最不可能越权**的一种；例如：BI 仅通过只读 HTTP/API 访问 1.2/2.0/2.1，不持有上游库写权限；所有“统计”均转发 2.0 或读 2.0 已持久化结果，不在 BI 层重算。
