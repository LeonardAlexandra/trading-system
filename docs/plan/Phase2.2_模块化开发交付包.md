## 一、总则

### 1.1 Phase 2.2 定位与只读边界（写死）

Phase 2.2 为**纯只读展示层（Read-Only Presentation Layer）**。BI 层职责仅限于**展示、对齐、解释、审计、运营观察**。以下宪法级约束必须在总则及**每一个相关模块的 Strong Constraints 中**反复体现：

1. **BI 层不写任何 Phase 1.2 / 2.0 / 2.1 的业务表**  
   不对 trade、decision_snapshot、execution、metrics_snapshot、evaluation_report、release_audit、learning_audit、strategy_version、param_version 等执行 INSERT/UPDATE/DELETE。若存在 BI 层缓存或物化视图，**仅**用于查询性能优化，且不改变上游业务语义。

2. **BI 层不调用任何会改变系统状态的接口**  
   不调用 Evaluator.evaluate；不调用 Optimizer.suggest 的写回路径；不调用 ReleaseGate 的 submit_candidate、confirm_manual、apply_approved、rollback_to_stable；不调用 MetricsRepository.write 或任何写入 2.0/2.1 表的接口。仅允许调用**只读**查询类 API。

3. **BI 层不新增计算口径、不派生第二套指标**  
   所有统计指标必须复用 Phase 2.0 MetricsRepository / evaluation_report 或通过 2.0 只读 API 获取；不在 BI 层重新定义计算公式。

4. **BI 层不生成「新解释」「新结论」「建议怎么做」**  
   决策过程仅展示 Phase 1.2 的决策输入快照与追溯结果；学习/版本/发布历史仅展示 2.0/2.1 既有字段；不合成「综合评分」「自动结论」。

5. **BI 层不成为控制面（无触发按钮、无状态变更）**  
   页面或接口中**不能**出现「在 BI 上点一下就生效」的按钮或调用（如「通过门禁」「执行回滚」「触发评估」「应用参数」）；仅可展示门禁/回滚/评估的**历史结果**。

### 1.2 推荐执行顺序（强制）

以下顺序为**推荐执行顺序**，不可调整。开发项必须按此顺序实施。

| 步骤 | 开发项 | 说明 |
|------|--------|------|
| 1 | A1 | 完整交易统计与资金曲线只读 API（数据来自 Phase 2.0） |
| 2 | A2 | 决策过程展示只读 API（数据来自 Phase 1.2 追溯与快照） |
| 3 | B1 | 版本与评估历史展示只读 API（数据来自 Phase 2.0） |
| 4 | B2 | 门禁/回滚/自动停用历史展示只读 API（数据来自 Phase 2.1） |
| 5 | B3 | BI 前端页面或模块（集成上述只读 API，无状态变更按钮） |
| 6 | B4 | 权限与脱敏（最小权限模型、脱敏字段、单租户/多租户约定） |
| 7 | D3 | 技术债专项修复：STABILITY（T2.2-TD-1） |
| 8 | D1～D4 | 端到端与回归可验证点（E2E-BI、E2E-BI-只读、E2E-BI-一致性、E2E-BI-决策过程与缺失） |

### 1.3 模块级执行规则（强制）

1. Phase 2.2 的开发必须严格按模块逐一推进。
2. 任一时刻，只允许存在**一个「活跃开发模块」**。
3. 当正在开发某一模块时：
   - 禁止修改、实现、重构任何非本模块定义范围内的代码；
   - 禁止提前实现后续模块的任何逻辑；
   - 禁止为「后续模块方便」而预埋代码、接口或占位实现。
4. 当前模块在**未通过验收**前，不得进入下一模块。
5. 若某模块验收失败，只允许在该模块范围内返工，不得牵连其他模块。

### A. 统计与曲线（数据来源：Phase 2.0）

#### A1. 完整交易统计与资金曲线只读 API（TBI-1）

**模块目标（Goal）**  
- 提供按日/周/策略的完整交易统计与资金/权益曲线**只读** API；数据**仅**来自 Phase 2.0 MetricsRepository、metrics_snapshot 或 2.0 只读聚合 API；供 BI 前端或下游只读消费。

**开发范围（Scope）**  
- **展示**：与 2.0 口径一致的 trade_count、win_rate、realized_pnl、max_drawdown、avg_holding_time 等；权益曲线时序数据；支持 from、to、group_by(day|week|strategy_id)、strategy_id（可选）。  
- **不展示**：不在本模块提供决策过程、版本历史、门禁/回滚历史；不提供任何会改变状态的入口。  
- **接口形态**（与蓝本 D.1 一致）：  
  - `GET /api/bi/stats?from=&to=&group_by=day|week|strategy_id`：返回盈亏、胜率、最大回撤、交易次数等；字段与 2.0 metrics_snapshot / evaluation_report 对齐；接口说明中显式标注**本 API 为只读，不改变任何业务状态**。  
  - `GET /api/bi/equity_curve?strategy_id=&from=&to=&granularity=day|week`：返回账户或策略维度的权益曲线；数据来源、时间粒度与聚合方式与 2.0 约定一致；显式标注**本 API 为只读，不改变任何业务状态**。  
- 多维度筛选与导出：仅增加筛选参数与导出格式，**不**改变统计口径；导出内容与接口返回一致。

**数据来源（Data Source）**  
- **仅**来自 Phase 2.0：MetricsRepository 只读查询、metrics_snapshot 只读查询、或 2.0 提供的只读聚合 API。**禁止**在 BI 层从 trade 表重新聚合计算；**禁止**直连 Phase 1.2 表做 BI 自有聚合。

**强制约束（Strong Constraints）**  
- **只读边界**：本模块不写任何 Phase 1.2/2.0/2.1 业务表；不调用 Evaluator.evaluate、Optimizer.suggest 写回、ReleaseGate 任何写或状态变更接口、MetricsRepository.write。  
- **口径复用**：所有指标与时间粒度、聚合方式**必须**与 Phase 2.0 B.2 及 C.1 一致；**禁止**在 BI 层新增或修改计算公式、**禁止**派生第二套指标。  
- 资金曲线/权益曲线数据来源与粒度在文档中写死并与 2.0 一致；**禁止**在 BI 层用自有公式重算「近似」权益曲线。

**验收口径（Acceptance Criteria）**  
- [ ] 可查完整统计（多维度 from/to/group_by/strategy_id）。  
- [ ] 可查资金/权益曲线（strategy_id、from、to、granularity）。  
- [ ] **数据一致性**：与 2.0 同条件查询结果一致（抽检）。  
- [ ] **只读**：该模块未调用任何写接口、未写业务表；验收可验证无写操作、无状态变更调用。  
- [ ] **口径**：未在 BI 层新增或修改计算公式。  
- [ ] 可选：大数据量下性能边界（如响应时间/分页）在文档约定。

**与蓝本验收任务绑定**  
- 覆盖蓝本 **TBI-1**、E 节 TBI-1 可验证断言。  
- 覆盖 **E2E-BI**（主流程中统计与曲线部分）、**E2E-BI-只读**、**E2E-BI-一致性** 中与本模块相关的验收点。

### A. 决策过程展示（数据来源：Phase 1.2）

#### A2. 决策过程展示只读 API（TBI-2 决策部分）

**模块目标（Goal）**  
- 提供单笔与列表的决策过程**只读**展示（signal→理由→风控→执行）；**仅**展示 Phase 1.2 的决策输入快照与追溯结果；不在 BI 层生成「新解释」或「推断原因」；PARTIAL/NOT_FOUND 时清晰展示 trace_status、missing_nodes、missing_reason。

**开发范围（Scope）**  
- **展示**：单笔决策链路（按 decision_id 或 signal_id）；决策列表（from、to、strategy_id、limit）；每条含 trace_status、missing_nodes（若 PARTIAL/NOT_FOUND）；字段与 1.2 追溯 API、决策快照 schema 一致。  
- **不展示**：不在本模块提供统计、资金曲线、版本历史、评估报告、门禁/回滚历史；不提供任何会改变状态的入口。  
- **接口形态**（与蓝本 D.2 一致）：  
  - `GET /api/bi/decision_flow?decision_id=|signal_id=`：返回单笔 信号→理由→风控→执行；来自 1.2 追溯 API；含 trace_status、missing_nodes（若 PARTIAL/NOT_FOUND）；显式标注**本 API 为只读，不改变任何业务状态**。  
  - `GET /api/bi/decision_flow/list?from=&to=&strategy_id=&limit=`：列表；每条可含 trace_status、missing_nodes；显式标注**本 API 为只读，不改变任何业务状态**。  
- 错误码：404 未找到、400 参数错误、503 上游不可用；敏感信息脱敏在 B4 约定。

**数据来源（Data Source）**  
- **仅**来自 Phase 1.2：追溯 API（按 decision_id/signal_id 或时间范围）、决策输入快照查询 API。**禁止**直连 trade/decision 表做 BI 自有聚合；**禁止**在 BI 层生成新解释或推断。

- **解释层**：仅展示 1.2 既有追溯与快照；**禁止**在 BI 层生成「新解释」或「应该怎么做」类推断；对 PARTIAL/NOT_FOUND **必须**清晰展示 trace_status、missing_nodes、missing_reason（或等价说明），不得静默隐藏。

**验收口径（Acceptance Criteria）**  
- [ ] 可查并展示决策过程（信号→理由→风控→执行）。  
- [ ] PARTIAL/NOT_FOUND 时展示 trace_status、missing_nodes 或等价说明。  
- [ ] **数据一致性**：与 1.2 同条件追溯/快照查询结果一致（抽检）。  
- [ ] **只读**：未调用发布/回滚/门禁/评估/学习接口；未写业务表。  
- [ ] 任一部分存在时 HTTP 200 且 body 含 trace_status；不存在的 decision_id/signal_id 返回 404 或 NOT_FOUND。

**与蓝本验收任务绑定**

**验收口径（Acceptance Criteria）**
