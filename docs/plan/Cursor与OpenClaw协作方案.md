# Cursor 与 OpenClaw 协作方案：后续开发任务分工与交接

**版本**: v1.2.0  
**日期**: 2026-02-22  
**依据**: Phase划分与实现顺序-需求清单、Phase2.0/2.1/2.2 模块化开发交付包、.cursor/rules/phase12-module-evidence.mdc

---

## 目的（引入 OpenClaw 的初衷）

1. **解放用户**：由 Cursor 与 OpenClaw 承担日常开发与交接，用户不必亲自指派每一模块、不必参与每次 handoff，只做**验收**与**关键范围/架构决策**。
2. **保证开发效果更好**：通过**固定顺序 + 证据包 + 开发/审计分离 + 单模块单主体**等约束，使双 agent 协作可追溯、可审计、少冲突，产出符合交付包与蓝本，质量不因「无人盯」而下降。

以下约定旨在让**双 agent 自协作、用户最小介入**的前提下，仍满足上述两点。

**OpenClaw 首次介入时**：请先读 **`docs/plan/OpenClaw介入指引.md`**，再读本协作方案与分工表，即可按同一套规则与 Cursor 协作，无需用户逐项说明。

## 一、如何连接、共同完成任务且不互相干扰

### 1.1 连接方式（推荐：文件系统共享 + 分工表）

- **当前采用**：**文件系统共享**。Cursor 与 OpenClaw 通过**同一仓库/同一目录**交换所有产出：代码、证据包、分工表、runlogs。无需额外 MCP 服务、API 或消息队列。
- **协调介质**：`docs/plan/当前活跃模块与分工表.md`（当前 Phase、当前模块、开发者/审计者角色、证据包路径）。谁完成本步谁更新该文件，另一方读该文件即知「该谁做、做啥」。
- **不互相干扰的保证**：
  - **角色分离**：每个模块**仅一方为开发者**（写代码、写证据包初稿），**另一方为审计者**（只读代码与证据包，对照交付包做审查，通过则签字/更新分工表，不通过则退回）。同一模块不同时由两人改代码。
  - **顺序与 scope**：严格按交付包顺序推进；开发者只改当前模块 scope 内允许的路径，审计者不改代码、只写审计结论或退回意见。
  - **单一真理源**：需求与验收口径以交付包与蓝本为准；实现与状态以证据包与分工表为准。双方不依赖私下约定或聊天记录。

### 1.2 可选增强（若需机器可读协调）

- 若希望工具或脚本自动解析「下一模块、当前负责人」：可增加 **`docs/plan/当前活跃模块与分工表.json`**（与 .md 同源维护），字段如 `current_phase`、`current_module`、`developer`、`auditor`、`evidence_pack_path`。仍为文件系统共享，只是多一份结构化状态，便于 MCP 或脚本读写。
- 若 Cursor 与 OpenClaw 运行环境**共享同一 MCP 客户端**（例如均作为 MCP server 注册到同一 Cursor）：可在此基础上增加 MCP 工具（如 `get_next_module`、`submit_evidence_pack`、`request_audit`），内部仍读写上述分工表与证据包路径，**仓库继续作为唯一事实源**。见下「连接方式对比与建议」。

## 二、角色分工：一方开发、一方审计（强制）

每个模块采用**开发 + 审计**分离，避免双写冲突并提升质量：

| 角色 | 职责 | 产出 | 禁止 |
|------|------|------|------|
| **开发者**（每模块仅一方：Cursor 或 OpenClaw，由分工表指定） | 按交付包实现该模块代码；运行测试/命令并记录真实输出；撰写证据包初稿（含文件清单、命令、输出、自检结论）。 | 代码变更、证据包初稿、可选 runlogs。 | 不得在未通过审计前进入下一模块；不得改 scope 外文件。 |
| **审计者**（另一方） | 只读该模块代码与证据包；对照交付包 Goal/Scope/Strong Constraints/Acceptance 逐项审查；可复现关键命令并核对输出；通过则更新分工表并注明「已审计通过」，不通过则写明退回理由（引用交付包条目）。 | 审计结论（写入证据包或单独审计备注）、对分工表的更新。 | 不得修改该模块实现代码；不得擅自放宽验收标准。 |

- **谁当开发者、谁当审计者**：在分工表或协作方案中写死约定（例如「奇数字母模块 Cursor 开发 / OpenClaw 审计，偶数字母反之」或「Phase 2.0 剩余 Cursor 开发 / OpenClaw 审计」）。双方仅按分工表执行，无需用户每模块指派。
- **用户**：仍为最终验收方；可在审计通过后再做抽检或阶段验收，也可委托审计者结论为通过依据。

## 用户角色（最小介入）

- **用户负责**：阶段/模块的**验收**（是否通过）；需求或范围的**变更决策**（是否改交付包/蓝本）；例外情况下的**仲裁**（若两方结论冲突）。
- **用户不负责**：每个模块「由谁做」的日常指派；逐模块的交接协调；阅读每一份证据包细节（可抽检）。  
- **实现方式**：由 Cursor/OpenClaw 通过**当前活跃模块与分工表**（见下）自驱推进；用户仅在阶段结束或抽检时查看证据包与验收结论即可。

## 四、协作原则（强制）

以下原则来自 `.cursor/rules/phase12-module-evidence.mdc` 与各 Phase 交付包，**Cursor 与 OpenClaw 均须遵守**。

1. **一次仅一个活跃开发模块**  
   任一时刻只有**一个**实现者（Cursor 或 OpenClaw）在开发**一个**模块；该模块未验收前不得进入下一模块。

2. **禁止跨模块修改**  
   只能改当前模块 scope 内明确允许的路径；越界修改视为失败，须返工。

3. **证据包是唯一有效交付物**  
   未生成或未更新证据包 = 任务未完成。证据包须含：模块名称与目标、修改/新增文件清单、实际执行的命令、命令真实输出、验收结论与逐条对照。

4. **顺序不可打乱**  
   Phase 2.0 → 2.1 → 2.2；各阶段内按交付包「推荐执行顺序」进行，不可跳过或调序。

5. **禁止并行开发同一模块**  
   Cursor 与 OpenClaw **不能**同时改同一模块或同一批文件；交接必须在该模块验收完成后。

6. **当前模块通过后始可进入下一模块（硬性门禁）**  
   **只有**当前模块经 **OpenClaw 审计通过**后，**Cursor 方可**进入下一模块的开发、测试或修改。  
   - Cursor 可**主动问询** OpenClaw 当前模块是否通过，或**等待** OpenClaw 回传/更新审计结论。  
   - 在未收到 OpenClaw 的「通过」确认前，Cursor **不得**开始下一模块的开发、测试或修改。  
   - 审计通过后，由 OpenClaw 更新分工表（或明确回传通过结论），Cursor 据此进入下一模块。

## 五、双 agent 自协作（无需用户逐次指派）

为减少用户参与，Cursor 与 OpenClaw 通过**同一份分工表 + 交付包顺序**自驱交接：

1. **当前活跃模块与分工表**  
   - 使用同一文件记录：当前 Phase、下一待做模块编号、当前负责方（Cursor 或 OpenClaw）、上一模块证据包路径。  
   - 推荐路径：`docs/plan/当前活跃模块与分工表.md`（可与本文档同目录，由任一 agent 在完成本模块后更新）。

2. **开发者（完成本模块实现后）**  
   - 写好证据包初稿并自检通过后，**更新分工表**：将本模块状态设为「待审计」，注明审计者（OpenClaw）；在证据包中写明修改/新增文件与下游依赖，供审计者读取。  
   - **审计通过前不得进入下一模块开发**：Cursor 须收到 OpenClaw 的通过确认（或主动问询 OpenClaw 并获通过）后，方可开始下一模块的开发、测试或修改。

3. **审计者（接手审查）**  
   - **先读**：`docs/plan/当前活跃模块与分工表.md` → 得知待审计模块、证据包路径、交付包中的 Goal/Scope/Acceptance。  
   - **再读**：该模块证据包与代码 → 对照交付包逐项审查，可复现关键命令；通过则更新分工表（本模块已通过审计、下一模块与下一开发者），不通过则写明退回理由。  
   - 不依赖用户口头或聊天指派；若分工表缺失，则按交付包顺序取「尚未有证据包的第一个模块」作为待开发模块，并依约定确定开发者/审计者。

4. **开发/审计约定（固定即可，用户不必每次说）**  
   - 在本文档或分工表中**写死**一种方式，例如：「Phase 2.0 剩余：Cursor 开发 / OpenClaw 审计」或「奇数字母模块 Cursor 开发 / OpenClaw 审计，偶数字母反之」。  
   - 两 agent 仅按该约定 + 分工表推进，用户无需每次指定。

## 六、推荐分工方式（写入分工表后由 agent 自动执行）

### 方式 A：按模块轮流（推荐）

- **约定**：当前模块的「实现者」仅一方（Cursor 或 OpenClaw）；该模块证据包完成后，下一模块可交给另一方。
- **示例**：  
  - Cursor 做 Phase 2.0 C3 → 交付证据包 → 用户验收。  
  - OpenClaw 做 Phase 2.0 C4 → 交付证据包 → 用户验收。  
  - Cursor 做 C5，OpenClaw 做 D1～D5 中的指定项（或反之），依此类推。
- **优点**：责任清晰，冲突最少；双方都熟悉交付包与证据包格式。
- **适用**：希望两边都参与同一 Phase、轮流推进时。

### 方式 B：按阶段切分

- **约定**：Phase 2.0 剩余由一方包干；Phase 2.0 全部验收通过后，Phase 2.1 由另一方（或同一方）按交付包顺序逐模块做。
- **示例**：  
  - Cursor 负责 Phase 2.0 剩余（C3、C4、C5、D1～D5）直至 2.0 封版。  
  - OpenClaw 负责 Phase 2.1 全部模块（A1～D6），严格按交付包顺序。  
  - Phase 2.2 再约定由谁做（或再轮流）。
- **优点**：阶段边界清晰，Phase 内上下文集中在一方。
- **注意**：Phase 2.1 必须在 Phase 2.0 全部验收通过后启动，不能提前并行。

### 方式 C：按类型切分（慎用）

- **约定**：一方负责「迁移 + 仓储 + 核心逻辑」，另一方负责「E2E/可验证点 + 证据包整理」。
- **风险**：交付包要求「当前模块未通过验收前不得进入下一模块」，若 E2E 与逻辑拆给两人，容易产生「逻辑已写、验收未过」的中间态，需严格约定「验收通过」的判定权在用户，且证据包由负责该模块的一方主写。
- **建议**：仅在双方都明确约定「谁负责哪几个具体模块编号」且不并行同一模块时使用。

## 七、交接规范

1. **真理源**  
   - 需求与范围：`docs/plan/Phase划分与实现顺序-需求清单.md`、各 Phase 的「模块化开发交付包」与「开发蓝本」。  
   - 实现与验收：各模块的**证据包**（`docs/Phase2.0_*_模块证据包.md`、`docs/phase1.2/Phase1.2_*_模块证据包.md` 等）。

2. **接手下一模块时**  
   - 以**交付包**中该模块的 Goal、Scope、Strong Constraints、Acceptance Criteria 为准。  
   - 以**上一模块证据包**确认「上一模块已验收、可进入下一模块」。  
   - **不依赖** Cursor/OpenClaw 的聊天记录；所有约定以文档与证据包为准。

3. **交接清单（建议每模块结束时提供）**  
   - 本模块证据包路径。  
   - 本模块修改/新增文件列表（与证据包一致）。  
   - 下游依赖（例如 C4 依赖 C3 的 evaluation_report 表与 Evaluator 接口）；若需下游注意的配置或约定，写在证据包或交付包附录中。

4. **命令与日志**  
   - 所有声称已执行的命令，其**真实输出**须在证据包中引用或写入 `docs/runlogs/*.txt`；禁止未执行却声称已执行。

## 八、冲突避免与禁止事项

| 禁止项 | 说明 |
|--------|------|
| 同时改同一模块 | 同一模块只允许一个实现者，验收完成后再交下一方。 |
| 跨模块修改 | 仅允许改当前模块 scope 内明确允许的路径。 |
| 跳过顺序 | 不得跳过交付包规定的执行顺序或 Phase 依赖（2.0→2.1→2.2）。 |
| 无证据包交付 | 未生成/更新证据包则视为任务未完成。 |
| 伪造执行 | 未真实执行命令却声称已执行，视为严重错误。 |
| 需求扩展 | 不得擅自解释、重定义或扩展需求与模块边界；变更须用户提出。 |

## 九、当前分工表与首次约定（供 agent 读取与更新）

- **分工表路径**：`docs/plan/当前活跃模块与分工表.md`（建议由首次启动的 agent 创建；之后由「完成当前模块」的 agent 更新）。
- **首次约定（写死一次即可，用户不必每次指定）**：在分工表或本文档中固定一种分工，例如：
  - **Phase 2.0 剩余**：按模块轮流（Cursor: C3、C5；OpenClaw: C4、D1～D5），或由 Cursor 包干 C3→C5→D1～D5。
  - **Phase 2.1**：Phase 2.0 封版后，按交付包顺序由 OpenClaw 包干 A1～D6，或轮流。
  - **Phase 2.2**：2.1 验收后按交付包顺序由 Cursor 或 OpenClaw 其一包干，或轮流。
- **重要**：分工表应包含「当前 Phase、下一待做模块、建议接手方、上一模块证据包路径」；agent 完成本模块后**必须**更新分工表并写好证据包，下一 agent 仅凭分工表 + 交付包即可接手，无需用户介入。

## 十、效果保障（为何能「开发效果更好」）

| 机制 | 作用 |
|------|------|
| 证据包强制 | 每模块可审计、可复现；用户抽检即可，不依赖记忆或聊天。 |
| 只读边界与单模块 scope | 减少误改、回归；Phase 2.0 不写 1.2，2.1 不写 2.0，2.2 纯只读。 |
| 固定顺序 | 依赖清晰，不会出现「下游先做、上游未就绪」。 |
| 开发/审计分离 | 每模块仅一方写代码、另一方只读并审查，避免双写冲突且多一层质量把关。 |
| 单模块单主体 | 同一模块不同时被两人改代码，避免合并冲突与行为不一致。 |
| 分工表 + 交付包为真理源 | 接手方不依赖对话记录，减少「传话失真」与遗漏。 |

用户只需在阶段结束时做**验收**（证据包 + 交付包逐条对照），即可在「解放自己」的前提下保持质量。

## 十一、连接方式对比与建议（回应 OpenClaw 提出的五种方式）

| 方式 | 与本项目的匹配度 | 建议 |
|------|------------------|------|
| **文件系统共享** | **最匹配、当前已采用**。双方共享同一 Mac 上的同一仓库；通过共享目录交换代码、证据包、分工表、runlogs。无需额外配置。 | **推荐作为主通道**。分工表 + 证据包即「任务与结果」的交换介质，天然避免同时写同一文件（约定：开发者写代码与证据包，审计者只写审计结论/分工表更新）。 |
| **MCP** | 适合结构化工具调用、任务协调。若 Cursor 与 OpenClaw 均接入同一 MCP 客户端，可暴露工具如 `get_next_module`、`submit_evidence_pack`、`request_audit`，内部读写分工表与证据包路径。 | **可选增强**。在保留「仓库为唯一事实源」前提下，用 MCP 做「谁该做啥、提交证据包」的标准化调用，便于自动化或第三方调度。不替代文件系统，而是基于同一套文件状态。 |
| **API 接口** | 灵活，但需常驻服务与鉴权。适合实时通信、状态监控。 | 当前阶段**不必选**。手off 为「做完→更新分工表→对方读表」的异步模式，无需实时 API；若未来需要「通知对方立刻拉取」可再考虑。 |
| **消息队列** | 异步解耦，适合长任务与工作流。 | 当前阶段**不必选**。任务与结果已通过证据包 + 分工表传递，队列增加运维成本且不解决「谁改哪文件」的冲突问题（仍依赖角色分离与顺序）。 |
| **直接进程通信** | 低延迟，但通常要求同机、同进程或固定父子关系。 | 取决于 OpenClaw 与 Cursor 的实际部署：若为不同产品/不同进程，**难以直接进程通信**；若 OpenClaw 作为 Cursor 内子 agent 运行，可由 Cursor 调度时传入「当前模块与角色」，仍建议以分工表为持久化状态。 |

**结论**：以**文件系统共享 + 分工表 + 证据包**为主连接方式，实现「连接且不互相干扰」；若需工具化或自动化协调，可在此基础上增加 **MCP 工具**或 **分工表的结构化版本（如 JSON）**，不引入额外服务或队列即可满足当前协作需求。
