## 零、架构级硬性设计约束

以下四条为全系统必须遵守的硬规则，各 Phase 交付包不得与之冲突。

### 0.1 学习边界（Learning Surface）

- **AI / Optimizer 仅允许修改「参数白名单」内的内容**。  
  **参数白名单**（与策略配置 schema 对齐，在《Phase 2.1 开发交付包》中与代码一致锁定）：
  - 策略配置中的数值型或枚举型**可调参数**：如 `max_position_size`、`fixed_order_size`、`stop_loss_pct`、`take_profit_pct`、信号权重或阈值类键（具体键名以当前策略配置 schema 为准，并在 2.1 交付包中显式列出「可学习参数清单」）。
  - 凡未列入「可学习参数清单」的配置键，Optimizer 与自动学习链路**禁止**写入或覆盖。
- **禁止修改的内容（不可变核心）**：
  - 策略的**执行逻辑与代码**（StrategyExecutor 的决策算法、分支与顺序）。
  - **风控核心**：RiskManager 的规则引擎、阈值来源配置（风控模块的启用/禁用及规则列表由配置管理，但风控逻辑代码与判定顺序不得由学习链路修改）。
  - **下单流程与幂等**：ExecutionEngine、OrderManager、decision_id 幂等、交易所适配层实现。
  - 信号接收、解析、去重、路由的**代码与协议**。
- **默认：人工确认（Human-in-the-loop）**。  
  策略或参数变更生效前，默认必须经**人工确认**（显式操作或审批流）。  
  **自动写回**（Optimizer 建议直接写回配置并生效）仅作为**可选能力**，且须在配置或特性开关中**显式启用**，并满足 0.3 发布门禁。

### 0.2 评估目标与约束版本化（Evaluator Contract）

- Evaluator 的**每次评估输出**必须包含以下**结构化、可持久化**内容（非仅统计报表）：
  - **目标函数定义**：本次评估使用的目标（如收益、夏普、回撤等）及权重或优先级。
  - **约束条件**：本次评估使用的约束（如最大回撤上限、最大风险暴露、最小交易次数等）及阈值。
  - **基线版本**：与之对比的基线（上一策略版本 ID，或历史时间窗口的标识）。
  - **结论**：通过/不通过或等级，及与基线对比的差异摘要（可机器可读）。
- 所有评估结果必须支持按**策略版本 ID / 评估时间 / 参数版本 ID** 查询；存储 schema 必须包含上述字段，不允许仅输出不可查询的报表文件。

### 0.3 发布门禁与回滚机制（Release Gate）

- **策略或参数变更上线前**，必须经过以下**至少一种**门禁：
  - **人工确认**：操作员显式确认该版本/参数可生效；或
  - **预设风控护栏校验**：系统自动校验该版本/参数在预设风控规则下是否允许生效（如回撤/风险暴露不超过阈值、或通过沙箱/回测结果校验），校验通过后方可进入「待生效」或生效。
- **必须支持**：
  - **一键回滚到上一稳定版本**：在运维界面或 API 上，可将当前生效的策略/参数版本回退到上一已标记为「稳定」的版本，且回滚后立即生效（或经同一门禁后生效）。
  - **自动停用异常版本**：当系统检测到满足「异常」条件时（如连续亏损超过 N 笔或 M 金额、回撤超过阈值、或系统健康检查失败），自动将当前策略置为暂停或回退到上一稳定版本，并产生告警与审计记录。异常条件与阈值的定义在《Phase 2.1 开发交付包》中明确列出。

### 0.4 决策输入快照（Decision Snapshot）

- **每次**生成交易决策（StrategyExecutor 产出 TradingDecision）时，系统**必须**持久化一条**结构化决策输入快照**，且与 decision_id 强关联、可追溯。  
- 快照至少包含（字段名与 schema 在《Phase 1.2 开发交付包》中与实现对齐）：
  - **行情/信号状态**：触发信号的关键字段（如 symbol、action、price、indicator_name、bar_time 等）、以及本次决策所用到的行情特征（若有时）。
  - **持仓状态**：决策时刻该策略下的持仓摘要（strategy_id、symbol、side、quantity 等）。
  - **风控检查结果**：本次决策前风控检查的通过/拒绝及原因（可引用 risk_check_id 或等价）。
  - **最终决策结果**：decision_id、symbol、side、quantity、reason 等。  
- 该快照为 **Phase 2.x 学习与评估的硬依赖**：Evaluator、Optimizer 与回放分析必须能按 decision_id 或时间范围读取决策输入快照，不得依赖「仅日志」或未结构化的文本。

## 一、Phase 总览与实现顺序

### 1.1 阶段列表与依赖关系

| 顺序 | Phase 名称 | 一句话目标 | 解决哪一层问题 | 前置依赖 |
|------|------------|------------|----------------|----------|
| 1 | **Phase 1.2a** | 可追溯 + 可运维：全链路追溯、决策快照、审计日志、监控告警与健康仪表板就绪。 | 可追溯、可运营 | Phase 1.0、1.1 已完成 |
| 2 | **最小 Dashboard** | 用户可在一个页面上看到最近决策/执行/成交与健康状态。 | 可运营 | Phase 1.2a 验收通过 |

| 4 | **Phase 2.0** | 可评估：按策略版本/时间产出带目标函数、约束与基线对比的评估报告，且可查询。 | 可评估 | 1.2a 契约满足（含决策快照） |
| 5 | **Phase 2.1** | 可学习、可发布：跑通「评估→参数更新→门禁→上线→再评估」至少一次，并具备回滚与自动停用异常版本。 | 可学习、可发布、可回滚 | Phase 2.0 完成 |
| 6 | **完整智能 BI** | 可审计、可运营：完整统计、决策过程与学习/变更历史在 BI 上可查可展示。 | 可审计、可运营 | Phase 2.0/2.1 数据与 API |

### 1.2 实现顺序（强制）

```
Phase 1.0（已完成）→ Phase 1.1（已完成）
  → Phase 1.2a：最小可追溯与可运维（含决策输入快照）
  → 最小 Dashboard（1.2a 验收通过后启动）
  → Phase 1.2b：全量审计与回放 + MVP 门禁验收
  → Phase 2.0：指标与评估底座（Evaluator 契约）
  → Phase 2.1：策略迭代与发布门禁、回滚
  → 完整智能 BI
```

### 1.3 与路线图共识的对应

- **1.2a 与 2.0 的契约**：1.2a 须满足「支撑 2.0 评估所需的最小审计与查询清单」且**必须包含决策输入快照**（0.4），满足后可衔接 Phase 2.0。
- **MVP 生产就绪门禁**：压力测试、故障恢复、备份恢复在 1.2b 收尾或 1.2a 与 1.2b 之间节点统一验收，不写入 1.2a 范围。
- **2.0/2.1 边界**：Phase 2.0 仅负责「可评估」（指标 + 带契约的评估报告）；Phase 2.1 负责「可学习、可发布」（参数更新、门禁、回滚、自动停用）。不允许单 Phase 同时承担「复杂强化学习」与「评估」的混合范围。

## 二、Phase 1.2a：最小可追溯与可运维

**正式名称**: Phase 1.2a：最小可追溯与可运维  
**一句话目标**: 系统具备全链路可追溯、决策输入快照落库、审计与错误日志可查、监控告警与健康仪表板，满足与 Phase 2.0 的契约（含决策快照）。  
**最小可验收闭环**: 从「信号接入」到「成交/拒绝」的任意一笔，均可通过 API 查到完整链路（signal → decision → 决策快照 → execution → trade），且决策快照字段符合 0.4；健康与告警可被 Dashboard 或监控消费。  
**完成判定标准**: （1）1.2a-0～1.2a-5 全部验收通过；（2）支撑 2.0 的最小审计与查询清单（含决策输入快照 schema 与查询方式）在交付包中显式列出且已实现；（3）至少一条端到端链路在测试环境验证「按 decision_id 可查决策快照 + 全链路」。

**目标（详）**: 建立全链路追溯能力、**决策输入快照**（见 0.4）落库与可查、审计/操作/错误日志入库与查询、监控与告警增强、简单健康仪表板、持仓一致性监控与报警；「任意时间点」首阶段降级为链路查询 + 当前/最近 N 笔状态；单链路回放以单笔 GET 串联形式实现。满足与 Phase 2.0 的契约后，可与 2.0 衔接。

### 开发项 1.2a-0：决策输入快照（Decision Snapshot）

**目标**: 落实 0.4：每次生成 TradingDecision 时持久化结构化决策输入快照，与 decision_id 强关联，供 Phase 2.x 评估与学习使用。

**涉及模块**:  
- StrategyExecutor 或决策产出点  
- 新增或扩展存储（DecisionSnapshot 表或等价），与《Phase 1.2 开发交付包》中 schema 一致

**关键接口/能力（需求级）**  
- 在产出 TradingDecision 的同一事务或原子流程内，写入一条决策输入快照。  
- 快照必含：行情/信号状态（symbol、action、price、indicator_name、bar_time 等）、持仓状态摘要（strategy_id、symbol、side、quantity）、风控检查结果（通过/拒绝及原因）、最终决策结果（decision_id、symbol、side、quantity、reason）。  
- 支持按 decision_id 查询单条快照；支持按 strategy_id + 时间范围批量查询。

**验收用例**  
- [ ] 每笔决策落库一条快照，decision_id 与 TradingDecision 一致。  
- [ ] 按 decision_id 可查得完整快照，字段满足 0.4。  
- [ ] 按 strategy_id + 时间范围可批量查询快照。  
- [ ] 追溯 API（1.2a-1）可返回「decision → 对应决策快照」或等价关联。

**交付物**  
- 决策快照 schema 与写入/查询接口（在《Phase 1.2 开发交付包》中锁定）。  
- 对「支撑 2.0 评估所需的最小审计与查询清单」的覆盖说明。

### 开发项 1.2a-1：全链路追溯 API 与单链路回放

**目标**: 提供按 signal_id / decision_id 的完整链路查询（signal → decision → **决策快照** → execution → trade），以及单笔「回放」（单接口返回整条链路），支撑 2.0 评估与最小 Dashboard。决策快照由 1.2a-0 提供并关联。

**涉及模块**:  
- 追溯查询服务或 API 层  
- 现有 TradeRepository、OrderRepository、决策/执行相关存储（或等价数据源）  
- 去重/幂等表（只读用于追溯）

**关键接口/能力（需求级）**  
- 按 **signal_id** 查询完整链路：返回该 signal 对应的 decision、execution、trade（单笔 GET 串联或单接口聚合返回）。  
- 按 **decision_id** 查询对应 execution 与 trade。  
- 按 **strategy_id** 的决策与成交列表查询，支持时间范围（起止时间）。  
- 按 **时间段**（起止时间）的决策与成交查询。  
- 「当前/最近 N 笔」状态查询（用于「任意时间点」降级）。

**验收用例**  
- [ ] 给定 signal_id，可返回完整链路（signal → decision → 决策快照 → execution → trade），数据与现有表一致。  
- [ ] 给定 decision_id，可返回对应决策快照、execution 与 trade。  
- [ ] 给定 strategy_id + 时间范围，可返回决策与成交列表。  
- [ ] 给定起止时间，可返回该时间段内的决策与成交列表。  
- [ ] 支持「当前/最近 N 笔」状态查询（N 可配置或固定上限）。  
- [ ] 以上能力可供内部服务或后续 HTTP API 使用（接口形态见《Phase 1.2 开发交付包》）。

**风险点**  
- 多表关联查询性能；数据缺失或未落库导致链路不完整。  
- 需与现有写入路径（ExecutionEngine、TradeRepository 等）对齐，避免漏写导致追溯断链。

**交付物**  
- 追溯查询模块或 API 实现（路径与命名见《Phase 1.2 开发交付包》）。  
- 1.2a 交付包中显式列出本开发项对「支撑 2.0 评估所需的最小审计与查询清单」的覆盖说明。

### 开发项 1.2a-2：LogRepository 审计/操作/错误日志入库与查询

**目标**: 将操作日志、审计日志（决策、风控、执行）、错误日志入库，支持按时间、组件、级别查询；不要求本阶段实现性能日志（延迟、吞吐量），性能日志放入 1.2b。

**涉及模块**:  
- LogRepository（扩展或新建）  
- 各业务模块的日志写入点（决策、风控、执行、异常）

**关键接口/能力（需求级）**  
- 写入：决策日志、风控检查结果、执行结果、错误/异常日志，可按时间、组件、级别写入。  
- 查询：按时间范围、组件、级别查询日志；支持分页或上限，避免单次拉取过大。  
- 审计日志至少覆盖：决策原因/结果、风控通过/拒绝及原因、执行提交/成交/失败。

**验收用例**  
- [ ] 关键操作（至少：信号接收、决策生成、风控检查、订单执行、成交/失败）有对应审计或操作日志入库。  
- [ ] 可按时间范围、组件、级别查询已入库日志。  
- [ ] 错误/异常有独立写入路径并可按级别查询。  
- [ ] 日志内容不包含敏感信息（如完整 API Key）；脱敏规则见《Phase 1.2 开发交付包》。

**风险点**  
- 日志量过大影响写入性能与存储；采样或分级策略在交付包中约定。  
- 与现有文件日志的边界：本项以「入库可查」为主，与文件日志的协同见《Phase 1.2 开发交付包》。

**交付物**  
- LogRepository 扩展或新表/新模块（见《Phase 1.2 开发交付包》）。  
- 日志写入与查询的调用约定或文档（见《Phase 1.2 开发交付包》）。

### 开发项 1.2a-3：SystemMonitor / HealthChecker / AlertSystem 增强

**目标**: 系统健康监控、组件级健康检查、告警规则配置与通知（邮件/日志），满足 7×24 可运维与「关键指标监控」的上线检查项。

**涉及模块**:  
- SystemMonitor（指标采集、健康度聚合）  
- HealthChecker（DB、交易所、策略状态等组件级检查）  
- AlertSystem（告警规则配置、分级、去重、邮件模板或日志）

**关键接口/能力（需求级）**  
- 指标采集：至少信号接收量、订单执行量、错误率等关键指标可采集并暴露（内存或可查询存储）。  
- 健康检查：DB 连接、交易所 API 连通性、策略运行时状态等可被轮询或主动查询。  
- 告警：支持配置化规则（如错误率超阈值、健康检查失败）；支持邮件或日志通知；支持简单去重与分级。  
- 关键指标监控（信号接收、订单执行、错误率）纳入本 Phase 验收，与 MVP 上线检查清单对齐。

**验收用例**  
- [ ] 关键指标（信号接收、订单执行、错误率）可被采集并查询或展示。  
- [ ] DB、交易所、策略状态等组件健康可被检查并返回明确状态。  
- [ ] 告警规则可配置（如阈值、接收人）；触发时能发送邮件或写入指定日志。  
- [ ] 日志可按时间/组件/级别查询（与 1.2a-2 协同，满足「日志查询和分析」上线项）。

**风险点**  
- 告警风暴；去重与冷却策略在交付包中约定。  
- 邮件依赖外部 SMTP；配置与失败降级见《Phase 1.2 开发交付包》。

**交付物**  
- SystemMonitor / HealthChecker / AlertSystem 增强实现（模块路径见《Phase 1.2 开发交付包》）。  
- 告警规则配置方式与示例（配置格式见《Phase 1.2 开发交付包》）。

### 开发项 1.2a-4：简单健康仪表板

**目标**: 提供 /health 之外的简单 Web 界面，展示系统健康、关键指标、最近告警等，便于运维一眼查看状态。

**涉及模块**:  
- 仪表板前端（简单 Web 页）  
- 后端：健康检查与指标查询 API（可复用 1.2a-3）

## 五、Phase 2.0：指标与评估底座

**一句话目标**: 系统具备「可评估」能力：按策略版本/时间产出带目标函数、约束与基线对比的评估报告，且评估结果可按策略版本 ID/评估时间/参数版本 ID 查询。  
**最小可验收闭环**: 对某一策略在某一时间窗口执行一次评估，得到一份**符合 0.2 Evaluator Contract** 的评估报告（含目标函数定义、约束条件、基线版本、结论与对比摘要），且该报告已持久化并支持按策略版本/时间/参数版本查询。  
**完成判定标准**: （1）2.0-1～2.0-5 全部验收通过；（2）Evaluator 每次输出均满足 0.2（目标函数、约束、基线、结论、可查询）；（3）至少一次在测试环境完成「指定策略+时间范围 → 产出报告 → 按版本/时间查询到该报告」的验证。

**目标（详）**: 可评估——基于现有 trade/decision/execution 与**决策输入快照**数据产出指标与评估报告；优先做实盘反馈闭环；历史数据管道在交付包中约定「历史数据假设」与硬性待办。

**前置条件**: 1.2a 契约满足（含决策输入快照）；1.2a 提供的审计与查询能力可用。

### 5.0 Evaluator 契约（硬性，见 0.2）

- Evaluator 的**每次**评估输出必须包含且持久化以下**结构化**内容（非仅统计报表）：
  - **目标函数定义**：本次评估使用的目标（如收益、夏普、回撤等）及权重或优先级。
  - **约束条件**：本次评估使用的约束（如最大回撤上限、最大风险暴露、最小交易次数等）及阈值。
  - **基线版本**：与之对比的基线（上一策略版本 ID，或历史时间窗口的标识）；若无基线则显式标注「无基线」。
  - **结论**：通过/不通过或等级，及与基线对比的差异摘要（可机器可读）。
- 评估结果存储 schema 必须包含：策略版本 ID、评估时间、参数版本 ID（若适用）、目标函数、约束、基线、结论等字段，并支持按**策略版本 ID / 评估时间 / 参数版本 ID** 查询。

## 六、Phase 2.1：策略迭代与增强学习

**一句话目标**: 系统具备「可学习、可发布」能力：至少跑通一次「评估 → 参数更新 → 门禁 → 上线 → 再评估」闭环，并具备一键回滚与自动停用异常版本。  
**最小可验收闭环**: 在测试或实盘环境中完成：**评估（2.0）→ 基于评估结果的参数更新（仅修改白名单参数）→ 经人工确认或风控护栏门禁 → 新版本生效 → 再次执行评估并可查询**；且支持一键回滚到上一稳定版本、以及（可选）自动停用异常版本。  
**完成判定标准**: （1）2.1-1、2.1-2、2.1-4 全部验收通过（2.1-3 按需）；（2）学习边界（0.1）在交付包中落实为「可学习参数清单」与禁止修改项清单；（3）至少一次完整闭环验证（评估→参数更新→门禁→上线→再评估）；（4）门禁、回滚与自动停用能力已实现并验收。

**目标（详）**: 可迭代——评估驱动参数/版本更新，闭环「实盘→评估→优化」；参数修改**仅限 0.1 参数白名单**，默认人工确认，自动写回须显式启用并过门禁。

**前置条件**: Phase 2.0 完成（或与 2.0 合并交付包、分里程碑验收）。

### 6.0 学习边界（硬性，见 0.1）

- **可修改**：仅限《Phase 2.1 开发交付包》中显式列出的「可学习参数清单」（与策略配置 schema 对齐，如 max_position_size、fixed_order_size、stop_loss_pct、take_profit_pct、信号权重/阈值等数值或枚举型可调参数）。Optimizer 与自动学习链路**禁止**写入或覆盖未列入清单的配置。
- **禁止修改**：策略执行逻辑与代码、风控核心逻辑与规则引擎、下单流程与幂等、信号接收/解析/路由代码与协议。
- **默认 Human-in-the-loop**：策略或参数变更生效前默认须人工确认；**自动写回**为可选能力，须显式启用且通过 0.3 发布门禁。

### 开发项 2.1-4：发布门禁与回滚（Release Gate，见 0.3）

**目标**: 落实 0.3：策略或参数变更上线前须经门禁；支持一键回滚与自动停用异常版本。

**涉及模块**:  
- 发布/上线流程或 API（人工确认或风控护栏校验）  
- 策略版本与「稳定版本」标记、回滚执行  
- 异常检测与自动停用（亏损/回撤/健康检查）

**关键接口/能力（需求级）**  
- **门禁**：策略或参数变更生效前，必须经过至少一种——（a）人工确认（操作员显式确认），或（b）预设风控护栏校验（如回撤/风险暴露不超过阈值、或沙箱/回测通过）。门禁通过后方可标记为「待生效」或生效。  
- **一键回滚**：运维界面或 API 支持将当前生效的策略/参数版本回退到上一已标记为「稳定」的版本，回滚后立即生效或经同一门禁后生效；回滚操作须留审计记录。  
- **自动停用异常版本**：当满足「异常」条件时（如连续亏损超过 N 笔或 M 金额、回撤超过阈值、系统健康检查失败），自动将当前策略置为暂停或回退到上一稳定版本，并产生告警与审计记录。异常条件与阈值在《Phase 2.1 开发交付包》中明确列出。

**验收用例**  
- [ ] 参数/策略变更生效前，至少经过人工确认或风控护栏校验之一。  
- [ ] 支持一键回滚到上一稳定版本，且回滚后可查询到回滚记录。  
- [ ] 异常条件与阈值已在交付包中定义；当触发时能自动停用或回滚并告警。

**交付物**  
- 门禁与回滚实现、异常条件与阈值文档（见《Phase 2.1 开发交付包》）。

## 十、完成 Phase 2.1 后的工程意义结论

**问：完成到 Phase 2.1 后，系统在工程意义上是否满足「可控的、自学习增强交易能力的交易员 bot」？**

**结论：是，在本文档与硬性约束（零、架构级硬性设计约束）被完整落实的前提下，可满足。**

- **可控**：学习边界（0.1）限定仅可修改参数白名单，核心逻辑与风控、下单流程禁止修改；默认人工确认，自动写回须显式启用且过发布门禁（0.3）；支持一键回滚与自动停用异常版本。  
- **自学习增强**：Phase 2.1 跑通「评估→参数更新→门禁→上线→再评估」闭环，Optimizer 仅产出白名单内参数建议，评估结果版本化可查（0.2），决策输入快照（0.4）支撑学习与评估。  
- **可回滚**：发布门禁与回滚（2.1-4）要求一键回滚到上一稳定版本及自动停用异常版本，并留审计记录。  
- **可审计**：决策输入快照、全链路追溯、审计日志、按版本/时间可查的评估结果，以及门禁/回滚操作记录，共同支撑可审计。

**若仍不满足时的可能差距（需在交付与验收时排查）**：  
（1）**参数白名单未真正锁定**：交付包中「可学习参数清单」未与代码与配置 schema 一致，或 Optimizer 可写入白名单外配置。  
（2）**Evaluator 输出未满足 0.2**：报告缺少目标函数、约束、基线或结论，或无法按策略版本/时间/参数版本查询。  
（3）**门禁或回滚未落地**：变更可绕过人工确认与风控护栏生效，或回滚/自动停用仅为文档约定而无实现与验收。  
（4）**决策快照缺失或不可查**：1.2a-0 未在每笔决策时落库或与 decision_id 未强关联，导致 2.x 评估与学习无法依赖。
