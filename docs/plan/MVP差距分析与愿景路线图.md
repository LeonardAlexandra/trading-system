- **1.2b（全量审计与回放）**：性能日志、多笔历史交易回放能力、审计日志查询界面（CLI 或 Web）、可选的状态快照或按时间还原（在 1.2a 验证通过后再定实现方式）。

## 三、愿景与系统定位（您提供的目标）

1. **数据与学习**  
   通过 TradingView 的 Indicator/Strategy 输出，用**历史数据做学习与预训练**，用**当前实盘数据做“增强”学习**，从而提升 AI 交易水平。

2. **Dashboard 与可观测性**  
   Dashboard 具备**完整智能 BI**：除完整交易统计外，能看到 **AI 决策、学习、决策变更** 的过程。

3. **一句话定位**  
   系统是「**7×24 小时在线、且会不断增强交易能力的 AI 量化交易员**」，用户能**看到 AI 操盘的全过程**。

**（共识修订）愿景与能力的对应关系**：为避免泛化并便于验收，正方在路线图中做如下澄清：  
- **「可观测与评估」**：日志、全链路追溯、指标计算（收益/回撤/胜率等）、评估报告、策略/参数版本查询——属「看到全过程」与「能评估效果」的能力，不要求自动改策略。  
- **「AI/学习」与「不断增强」**：在本系统中指基于实盘与历史数据的**评估结果**驱动**人工或半自动**的策略参数/版本调整或权重更新；后续可扩展为自动优化（Optimizer）、晋升淘汰（PromotionEngine/EliminationEngine）等，具体形态与权限边界在 Phase 2.x 交付包中定义。  
- **「不断增强」的可验收表述（示例）**：每 N 周可产出策略/周期评估报告；支持基于评估结果进行一次参数或策略版本的更新（人工确认或半自动）；指标历史可查、可对比。具体 N 与自动化程度在 Phase 2 规划时敲定。

## 四、差距与愿景的对应关系

- **Phase 1.2（日志与可追溯）**  
  - 为「看到 AI 操盘全过程」打基础：  
    全链路追溯（信号→决策→执行→成交）、审计日志、回放、健康与告警，都是「过程可见、可查、可复盘」的前提。  
  - 建议：在推进「AI 学习与 BI」前，先完成 Phase 1.2a 与最小 Dashboard（详见第五节），避免过程黑盒；1.2b 可与 2.0 衔接或稍后收尾。

- **MVP 上线检查清单**  
  - 保证当前系统可稳定 7×24 运行、可运维、可恢复，是「在线 AI 交易员」的底座。  
  - 建议：在宣称「MVP 生产就绪」或进入 Phase 2 前，逐项验收并补齐。

- **评估与优化层 + Shadow/Simulator**  
  - 对应「历史数据学习 + 实盘增强学习」：  
    需要指标计算与存储、评估器、优化器，以及影子/模拟环境，才能闭环「学习→决策→执行→反馈→再学习」。  
  - 建议：列为 Phase 2.x「AI 学习与策略迭代」的核心范围。

- **Dashboard 与智能 BI**  
  - 当前无正式 Dashboard；MVP 仅提到「简单 Web 健康仪表板」。  
  - 「完整交易统计 + AI 决策/学习/决策变更过程」属于**新增产品范围**，需单独规划（前端 + 统计/指标 API + 决策与学习过程的数据模型与存储）。

## 五、建议实现顺序（与愿景对齐）

1. **Phase 1.2a：最小可追溯与可运维（推荐优先）**  
   - 全链路追溯 API、审计/操作/错误日志入库及查询、监控与告警增强、简单健康仪表板、持仓一致性监控与报警；「任意时间点」首阶段降级为链路查询 + 当前/最近 N 笔状态。  
   - 与 1.2 重叠的上线检查项（关键指标监控、日志查询）在本阶段一并验收；压力测试、故障恢复、备份恢复在 1.2 规划中明确时机。

2. **最小 Dashboard（与 1.2a 收尾或 2.0 初期同步）**  
   - 决策/执行/成交列表（按时间或 signal_id 查）、简单汇总（笔数、盈亏、按日或按策略）、健康与关键指标展示；与追溯 API 共用数据源。  
   - 满足「看到 AI 操盘全过程」的最小可读版本，再在 Phase 2.x 迭代为「完整智能 BI」。

3. **Phase 1.2b（在 1.2a 与最小 Dashboard 可用后推进）**  
   - 性能日志、历史交易回放、审计日志查询界面、可选的状态快照或按时间还原。  
   - 收尾 MVP 上线检查清单其余项，达到「MVP v1.0 生产就绪」。

4. **Phase 2.0：指标与评估底座**  
   - **主线**：可评估——基于现有 trade/decision/execution 数据产出指标与评估报告。  
   - MetricsRepository、MetricsCalculator、Evaluator；策略版本与评估结果可查。  
   - **优先做实盘反馈闭环**。历史数据管道：在 2.0 交付包中增加「历史数据假设」一节，硬性待办与触发条件见第八节；2.1 结束前须约定格式、至少一条导入路径及责任方。

5. **Phase 2.1：策略迭代与增强学习**  
   - **主线**：可迭代——评估驱动参数/版本更新，闭环「实盘→评估→优化」。  
   - Optimizer（参数优化/回测框架）、实盘反馈写入指标与评估；ShadowExecutor / MarketSimulator、PromotionEngine、EliminationEngine 按需纳入。  
   - 「预训练」与「增强学习」的具体形态（离线回测评估 vs 在线参数微调）在 Phase 2.x 交付包中定义。

6. **完整智能 BI（独立产品线，与 2.0/2.1 数据对接）**  
   - 交易统计、资金曲线、胜率/回撤、决策过程（信号→理由→风控→执行）、学习与决策变更（参数/版本、评估与优化历史）。  
   - 数据模型、API、前端与权限单独规划。

### 七.1 正方的质疑与保留（批判性回应）

正方对反方建议并非全盘接受；以下为**质疑、条件或反向风险**，供反方与后续讨论时权衡。无对错之分，仅呈现另一侧视角。

- **关于 1.2 拆段与 1.2b 后置**  
  **质疑**：若 1.2a 交付后立刻启动 2.0，而 1.2b（性能日志、回放、审计查询界面）长期搁置，则 2.0 的评估可能缺少「延迟/吞吐」等维度，且事后排查仍依赖「回放」能力，缺口会一直存在。  
  **保留**：只有在「1.2a 验收标准明确包含足以支撑 2.0 评估的审计与按维度查询」的前提下，正方才接受 1.2b 后置；否则仍倾向在进入 2.0 前至少完成「审计入库 + 追溯 API + 单链路回放」，再开放评估能力。

- **关于最小 Dashboard 与 1.2a 同步**  
  **质疑**：若最小 Dashboard 与 1.2a 严格同步开发，会分散 1.2a 精力（前端、API 设计），可能拖慢「可追溯」与监控本身；且若 Dashboard 先上而追溯 API 未稳定，用户看到的是不完整或易变数据，体验反而差。  
  **保留**：正方更倾向「1.2a 验收通过后，再启动最小 Dashboard」，或明确「Dashboard 依赖 1.2a 追溯 API 稳定版」；若资源紧张，可接受 Dashboard 再后移一档（2.0 初期），优先保证后端可追溯与监控稳定，而非为「同时交付」牺牲 1.2a 质量。

- **关于历史数据长期后置**  
  **质疑**：若过度强调「先做实盘、历史后置」，长期会形成只有实盘一段数据的评估，缺乏与历史回测的对比，策略迭代的置信度会打折扣；愿景明确提到「历史数据做学习与预训练」，若 Phase 2 完全不碰历史，该表述会长期悬空。  
  **保留**：同意 2.0 主线为实盘反馈闭环，但要求在 2.0 或 2.1 交付包中**明确写入**「历史数据管道的最低范围与触发条件」（例如：2.1 结束前必须完成历史数据格式约定与一条导入路径），避免历史能力被无限期后置。

- **关于压力测试、故障恢复、备份恢复并入 1.2**  
  **质疑**：若将压力测试、故障恢复、备份恢复都写进「1.2 规划」且与 1.2a 绑定，1.2a 范围会再次膨胀；且压力测试依赖日志与监控就绪，若在 1.2a 未完全可观测时做，结论代表性不足。  
  **保留**：正方同意「关键指标监控、日志查询」纳入 1.2a 验收；但「压力测试、故障恢复、备份恢复」更倾向单独列为 **MVP 生产就绪门禁**，在 1.2a 与 1.2b 之间或 1.2b 收尾时统一验收，而非硬性写入 1.2a 范围，避免 1.2a 边界模糊与延期。

- **关于 Phase 2.0/2.1 拆死**  
  **质疑**：2.0（指标与评估）与 2.1（策略迭代）可能强耦合（如 Evaluator 输出直接驱动 Optimizer），若在路线图层面就把两阶段拆死，后续交付包可能发现重复开发或接口反复。  
  **保留**：同意在路线图中给出 2.0/2.1 的主线与边界，但保留「2.0 与 2.1 可合并为一个交付包、分两个里程碑验收」的弹性，待 Phase 2 启动时根据实际依赖再定，避免为拆分而拆分。

- **关于「AI/学习」「不断增强」的收敛表述**  
  **质疑**：若在文档中把「不断增强」过早收敛为「每 N 周报告 + 人工/半自动更新」，可能限制后续对「自动调参、多策略择优」等形态的想象；且「AI」在对外表述上仍有品牌与用户预期价值，不宜在路线图中收得过窄。  
  **保留**：同意在**内部验收**层面使用「可观测与评估 vs 自动优化/学习」的区分及可验收表述；但对**对外愿景表述**保留「7×24、会不断增强、看到全过程」的开放性，不在路线图中承诺「不做全自动优化」或「只做半自动」，以便后续按产品选择扩展。

- **关于收缩选项的表述**  
  **质疑**：若在路线图中写死「资源不足时 1.2 仅做 1.2a」，可能给决策者「1.2b 可无限期推迟」的暗示，回放与全量审计永远排不上优先级；且收缩选项过多，可能削弱对「完整交付 1.2」的承诺。  
  **保留**：同意增加假设与收缩选项，但建议在文档中同时写明「**收缩为临时应对，目标仍是完整 1.2 与 MVP 生产就绪**」；且收缩后的「最小可运行形态」需明确（例如：无回放、无性能日志），以便后续补课时有清晰 backlog，而非默认长期停留在收缩态。

## 八、假设与风险、已明确事项

**（共识）假设与风险（简述）**  
- **迭代节奏**：假定按 Phase 分阶段交付，每阶段有明确验收与可选的收缩口径。  
- **外部依赖**：实盘依赖交易所 API 与 TradingView Webhook；历史数据依赖 TradingView 历史导出方式（见下「历史数据管道」已明确）。若历史数据不可用或滞后，优先保证「实盘→评估」闭环，历史管道后置或降级。

**（已明确）原悬置三项——正方回复与落地方案**  
- **1.2a 与 1.2b 的正式命名与交付包拆包**：采纳反方建议。正式命名为 **Phase 1.2a：最小可追溯与可运维**、**Phase 1.2b：全量审计与回放**；在《MVP 实现计划》修订版或单独《Phase 1.2 开发交付包》中成文，各自对应验收标准（可同文档内两章），与 Phase 1.0/1.1 交付包形态一致。  
- **「最小可评估」与 2.0 的边界**：明确为——不单独增设 Phase 1.5。1.2a 验收通过后，若已具备「按 strategy_id/时间段的决策与成交查询」及 1.2a 契约清单（见第九节），则**直接进入 Phase 2.0**（MetricsCalculator + Evaluator）；若实施时希望更轻量探路，可先做「只读现有 trade/decision、产出简单汇总指标（笔数、盈亏）的脚本或单接口」作为 2.0 探路，不纳入正式 Phase 命名，探路完成后再进入 2.0 正式交付。以「直接 2.0」为默认路径。
