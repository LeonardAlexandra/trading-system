## 一、历史数据假设

### 1.1 责任方定义
- **历史导出方（数据源提供）**: 
  - 默认由策略运营方或 TradingView 账户持有人负责从 TradingView 图表或策略测试器导出历史成交记录。
  - 若使用自动化工具，则由工具维护者确保导出数据的准确性。
- **落地导入方（系统接入）**: 
  - 由系统开发者（或指定的 ETL 工程师）负责将导出的历史数据映射并导入至系统 `trade` 表。
  - 需确保导入数据不触发重复信号告警，且正确标记 `source_type`（如标记为 `HISTORY` 或 `BACKTEST`）。

### 1.2 TradingView 侧可行方式
- **CSV 手动导出（推荐/首选）**: 
  - 通过 TradingView "Strategy Tester" (策略测试器) 的 "List of Trades" (交易列表) 导出 CSV 文件。
  - 这是最简单、低成本的离线获取方式。
- **手动录入**: 
  - 针对极少量关键成交记录，可手动构建 SQL 插入脚本或通过管理界面录入。
- **API/脚本抓取（备选）**: 
  - 若 TradingView 提供相关 API 或通过第三方插件，可考虑自动化抓取，但本阶段不做实现要求。

### 1.3 与实盘 Webhook 格式的一致性要求
- **核心字段对齐**: 导入的历史数据必须包含实盘 Webhook 触发后产出的 `trade` 表核心字段：`strategy_id`、`symbol`、`side`、`quantity`、`price`、`executed_at`。
- **计算口径一致**: 历史数据的 `realized_pnl` 计算逻辑必须与 Phase 1.2 `TradeRepository` 及 Phase 2.0 `MetricsCalculator` 的口径保持一致，确保评估结果的可比性。
- **版本关联**: 导入记录应能关联到对应的 `strategy_version_id`，若历史导出中无版本信息，需在导入时手动补全或默认关联至历史基线版本。

## 二、硬性待办（Hard Todos）

在 **Phase 2.1 结束前**，必须完成以下约定的确认：

1.  **历史数据格式标准**: 明确 CSV 文件的列头定义（Header Mapping）及日期格式标准。
2.  **至少一条导入路径**: 确定是采用“后端管理接口导入 CSV”还是“数据库直接跑批导入”作为标准路径。
3.  **最终责任方确认**: 正式指定负责历史数据定期更新与导入的岗位或人员。
4.  **历史数据校验规则**: 约定如何验证导入数据的完整性（如总笔数核对、盈亏汇总校验）。

---

## 三、触发条件

启动历史管道（History Pipeline）的触发条件：
- **条件 A**: 实盘评估闭环（Phase 2.0 C1~C4）已通过 E2E 验证并稳定运行。
- **条件 B**: 需要将当前实盘表现与历史长期回测表现进行量化对比，以验证策略稳健性时。
- **条件 C**: 策略发生重大版本变更，需追溯历史表现作为新版本基线（Baseline）时。

---

## 四、Phase 2.0 边界声明

本模块及本阶段（Phase 2.0）的实现边界严格界定如下：
1.  **不实现导入代码**: 本仓库中禁止出现任何用于解析 CSV、连接外部数据源导入历史数据或执行批量导入的脚本、任务（Job）或接口实现。
2.  **仅交付文档**: 本模块的唯一交付物为本「历史数据假设与硬性待办」文档。
3.  **不改动 Schema**: 不为历史导入需求在数据库中预埋任何字段或表结构。
4.  **无自动逻辑**: 不引入 Phase 2.1 涉及的参数写回、Optimizer、自动学习或自动决策修改等高级能力。
